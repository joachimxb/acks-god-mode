# Changelog

All notable changes to ACKS God Mode are documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/),
and this project adheres to [Semantic Versioning](https://semver.org/).

## [Unreleased]

### Changed
- **A consistent parchment palette across the interface.** The app's colour system is now the single source of its colours: the redundant hues that had crept in for the same meaning are unified — one green for success (the two near-identical greens collapsed to the palette token), one amber for caution (the stray oranges folded in), one red for danger (the stray roses folded in), and the cool greys that leaked onto the warm parchment ground warmed to match — both on the main screen and in the feature panels. The universal pale-yellow hover (used on roughly 280 rows, tabs, and buttons) is now one shared hover tint, and roughly 600 hand-styled buttons adopt the shared button style: the plain ones size and behave alike, and the coloured ones — confirm, delete, caution, and primary actions, including every wizard's submit button — take a matching role colour (green border for confirm, red for delete, gold for the primary call-to-action) so the same kind of action looks the same everywhere, while still greying out when their form isn't ready. The success-green, caution-amber, and danger-red used for status text, panel tints, and borders now draw from those same palette tokens rather than scattered raw colour shades, so every "this went well / take care / this is dangerous" cue is the one agreed hue. The dozens of hand-mixed one-off panel and badge tints that had drifted from the palette are nudged onto the nearest agreed role colour too, so a good / neutral / bad panel reads in the same green / parchment / red across the whole app — a subtle shift on some warm panels, and the one genuinely visible refinement in an otherwise like-for-like cleanup. The last few raw colours that had stayed outside the palette — the faint rule line in the bookkeeping tables and the purple accent on the magic-research panels — now draw from it too, so nothing on the surface is a hand-typed colour any more. Separately, every pop-up dialog now draws its card — parchment background, border, and shadow — from one shared modal style rather than each dialog hand-styling its own, and the dark scrim that dims the page behind a dialog (previously set three different ways) draws from one shared backdrop tint too, so the dialogs read consistently (a few heavier-bordered newer dialogs settle to the common look, and the wide setup wizards keep their distinct wider card and a slightly heavier scrim). The tall scrolling dialogs likewise take their full-screen frame — positioning, top alignment, scroll, and the page-dim — from one shared overlay style instead of each repeating the same handful of layout classes (the stacking order of overlapping dialogs is unchanged). A build-time check now keeps the retired colour families and the modal styles from drifting back in. Purely visual — no rules, data, or save behaviour changes; the look is more uniform, nothing moves. (Graphic-design audit H1 — the colour-token sweep and the modal-surface standardization; the emoji→SVG icon system is now underway, below.)
- **Printed ledgers and sheets look right now.** Printing the app (the domain ledgers, character sheets, and the tables GMs hand out or file) was previously near-unstyled — backgrounds were forced white and everything else was left to the browser. It now carries real print rules: a long ledger repeats its column headers at the top of every printed page, a row is kept whole rather than split across a page break, and the faint table rules darken to ink so they survive on white paper. On-screen, nothing changes. (Graphic-design audit H1 — print.)
- **Clearer heading hierarchy, and no more tiny text.** Section headings and the sub-headings beneath them used to render at the same size, so the page structure was hard to scan; section headings now sit one step larger than their sub-headings, giving a readable title → section → sub-section ladder. Separately, the few text snippets that had been shrunk below the interface's smallest standard size are lifted back to it. (Graphic-design audit H1.)
- **A crown wordmark in the header.** The app's crown mark — the one in the browser-tab icon — now sits beside the "ACKS God Mode" title, drawn in the parchment palette, so the header carries the product's identity rather than plain text alone. (Graphic-design audit H1 — identity.)
- **Crisper icons that match the theme.** The interface's recurring identity icons are migrating from coloured emoji to inline SVG drawn in the parchment palette. Two benefits: they render *sharply and identically on every platform* (emoji are drawn by the operating system and vary — and look out of place against parchment), and they *inherit the surrounding text colour*, so the same icon goes cream on a dark button, ink on a panel, and red on a danger control — always matching its context. The glyphs converted so far are the **edit pencil** beside every in-place editable value, the **search / inspector** mark, the **chronicle / events** scroll, and the **domains / stronghold** mark; the rest of the recurring set (combat, treasure, money, the navigation tabs, …) follows. Mentions of an icon inside a sentence, help text, and the purely decorative one-offs stay as emoji. Purely visual — the edit buttons keep their behaviour, and gained proper screen-reader labels where a bare emoji had been their only name. (Graphic-design audit H1 — the emoji→SVG icon system; rolling out glyph by glyph.)

## [0.51.0] - 2026-06-26

### Added
- **A bleeding treasury now has consequences (RR p.349).** When a domain ends a month **insolvent** — its treasury actually in the red, not merely running a deficit it can still cover from reserves — its ruler could not pay the garrison that month, so the realm's **next domain-morale roll takes a penalty**: −1 the first insolvent month, deepening by 1 for each consecutive month the coffers stay empty (to a maximum of −4), and clearing the first month you are back in the black. Sustained insolvency therefore drags morale down — which collapses income (a sullen populace works less) and raises bandits from the unpaid and the disaffected (RR p.351). An empty treasury is no longer just a number that falls forever; it sets a realm sliding toward unrest. A wealthy domain absorbing a lean month from its reserves is untouched — the penalty bites only once the reserves are truly gone.

### Changed
- **Domain banditry is now always-on core RAW (RR pp.350–351).** A domain that falls to Turbulent morale (−2) or worse is plagued by bandits — its own able-bodied men taking to the roads (one per five families at −2, per two at −3, all of them at −4) — and this now always applies, exactly as the rulebook intends. It was previously gated behind a default-on house rule that could be switched off; that toggle is gone (see Removed).

### Fixed
- **World Import / Export works on the hosted site again.** The 📊 World Import / Export box (and the ⬇ Template / ⬇ Export world / ⬆ Import buttons) failed on the live site with *"Could not load the spreadsheet library (xlsx) from the CDN"* — the app's security policy (Content-Security-Policy) blocks any script loaded from outside the app, and the spreadsheet engine was being fetched from a CDN. The library is now **bundled with the app** (loaded on demand from the app's own files, exactly like Alpine and Tailwind), so export and import work on the hosted site and fully offline. Same pinned build (SheetJS 0.18.5), integrity-verified, still loaded only when you open the box.
- **Domain morale — bandits "count as an enemy army" more faithfully (RR pp.349–354).** The cumulative enemy-army occupation penalty that unchecked bandits inflict on the monthly morale roll now correctly **caps at −4** (RR p.352; it previously kept deepening past −4 month after month). And when a bandit lord is actively **pillaging** the domain, his −4 penalty now applies *in lieu of* the occupation penalty rather than stacking on top of it (RR p.354 — the two do not stack).

### Removed
- **The `domain-morale-banditry` house rule.** Domain banditry is core RAW (RR pp.350–351) and always applies, so the opt-out toggle has been removed. Existing saves that set it are unaffected — the stale flag is simply ignored.

## [0.50.0] - 2026-06-25

### Added
- **Let the garrison answer small raids on its own (JJ p.104).** A domain's **⚔ Active Threats** box (Domains ▸ Military) gains a checkbox — **🛡 Garrison auto-defends low-threat incursions**. Tick it and whenever an incursion band wanders in that the garrison can clearly handle — its battle rating is **at least twice the band's** — the garrison **musters a reaction force the same day it appears, then marches out the next day to meet it on its own**, exactly as if you'd ordered the sally by hand. (A standing garrison is already gathered at its stronghold, so it forms up in a day — not the weeks RAW reserves for raising or calling up scattered troops; RR p.434, JJ pp.103–104. The manual ⚔ Deploy sally musters the same one day.) The band **stays on the threats list, marked 🛡 Responding** (with the force's status — mustering, marching to engage, in position), while the troops assemble and close — you can still watch the force on the 🎖 Armies card and re-route or recall it — until it arrives, **drives the band off** (taking light, proportional losses: a sliver at the 2× margin, nothing when it overwhelms the band), and marches home. The garrison **commits only as many units as a decisive drive-off needs**, not the whole muster — so a multi-unit garrison can **answer several bands at once**, and any threat it couldn't reach when it first appeared (its troops were already out, or its scouts hadn't yet spotted the band) is **picked up on a later day** as a sortie returns and re-garrisons. A band the garrison *can't* clearly beat — a near-match, or a dragon the rules won't price — still waits for your orders, exactly as before. By default the garrison sallies only against **hostile or unfriendly** bands; a second checkbox — **⚔ Also drive off neutral bands** — extends it to neutral/exploratory bands too (the act of marching on one turns it unfriendly, JJ p.104). Left off, the garrison lets a neutral band pass (and an unchallenged neutral band leaves the peasants uneasy — the −1 noted below). The box now shows even when there are **no** active threats, so the checkbox always has a home. Off by default — a per-domain switch, not a house rule; each ruler decides whether his garrison acts without him. A garrison out on such a sortie **within its own domain still counts toward the realm's security** (the morale/garrison-adequacy check — RR p.341, "to maintain the security of his domain"; a garrison that has *left* the domain does not — RR p.169, JJ p.104). While it defends its own land the sortie is **fed by the realm** — it draws no campaign supply and is never mistaken for an army occupying its own domain (a garrison is upkept by its garrison cost, RR p.341, not a baggage train); and once the band it answers **leaves the domain**, the garrison **stands down and holds its ground** rather than chase it off into the wild — a garrison defends its domain (RR p.341), it doesn't go on campaign. This holds even when a migrating band **wanders off the edge of the mapped hexes** (the garrison judges where the band has gone by its true position, not by whether it's still on an authored hex — so it never marches out after a band it can no longer reach), and such a band shows **its coordinates** in the 🐉 Monsters ▸ Groups list (e.g. "near 30,30 — off the mapped hexes") instead of a bare "no fixed hex". And if a band **digs in and makes a lair** in the time it takes the reaction force to reach it, driving it off now **clears out that fresh den too** — the lair is left abandoned (a standing site that can be re-occupied later), never an empty lair still pointing at a band that's been chased off the map. And the ruler now **learns of an incursion the way the rules say he does** (JJ p.103, RR p.452): a band is spotted **at once** if it reaches the **stronghold** or is caught **pillaging a settlement**, and otherwise the ruler's **reconnaissance is re-rolled each week** — and **every day** while the band lurks within one hex of any of the domain's settlements — so a band that slipped past the *first* scouting check at its arrival is no longer invisible forever. (A band merely lingering in the deep wilds can still go unnoticed for a time — settling, even gnawing at the domain's morale, while an oblivious ruler hunts for the cause — exactly as JJ p.103 allows; what's fixed is that a failed arrival check no longer blinds him *permanently*.) The instant the garrison's scouts locate a band, an auto-defending domain sorties against it that same day. And the threat list now reckons with **every monster physically in the domain**, not just the ones the abstract roll conjured: a band that **wanders across the border** is treated as a full domain encounter (it rolls a disposition + the ruler's reconnaissance, lands in ⚔ Active Threats, and a neutral one that crosses unchallenged leaves the same peasant unease); and a **monster that has denned (laired)** in the domain's hexes — whether it settled there after arriving or is an ambient wilderness lair that was always there — now appears as a standing threat too, with its own awareness state. Once the ruler's scouts have located such a den, the garrison will **clear a weak one on its own** (the same decisive-advantage rule it uses to drive off a field band), while a den it can't handle — or one it hasn't found yet — waits, as ever, for you. The **bundled demo ships with it on for every domain**, and the demo's marches are now garrisoned and strongholded to their RAW minimums (and their borders judged secure) so an unattended realm's incursions stay at a pace the garrisons can keep in check rather than flooding the frontier. (D4 — garrison auto-defense; the slot-89 day consumer deploys the shipped reaction-force flow over the JJ pp.104–106 drive-off + the platoon-BR comparison.)
- **Bulk world import/export — author your world in a spreadsheet (`.xlsx`).** A new **📊 World Import / Export** box at the top of the **World** tab (above the World sub-tabs, visible across all of them) round-trips your world-state through a real Excel workbook — one sheet per entity (**Domains · Hexes · Settlements · Lairs**), one row per entity. **⬇ Template** downloads a blank workbook (with a generated read-only *Reference* sheet listing every legal terrain, Köppen code, climate, classification, lair type… so the values can't drift), **⬇ Export world** writes your current world out, and importing an `.xlsx` shows an **inline preview** — per-sheet tallies (new / update / skip), a **Create-only / Update-existing** toggle, and a per-row error+warning list — before a single **Commit** (nothing changes until you approve, and bad rows skip while good rows land). Coordinates are plain Col/Row columns, references resolve by name *or* id, families live on the domain (the revenue-authoritative figure) while a hex's classification derives from its owner — so the sheet obeys the same rules the engine does. Built on a new **`hex` + `domain` field-schema** (the single description the importer, the Inspector edit form, and a future Hex/Domain Wizard all read — so they can never disagree about what a hex *is*; this also advances the Inspector's hex/domain Admin-Create, Wave-F). SheetJS loads on demand from a pinned CDN only when you open the box, so the app's no-build, three-script-tag delivery is unchanged. The old Domains-tab "📥 Import domains…" JSON merge is **retired** — whole-campaign sharing stays on **📂 Open campaign**; world-skeleton sharing is the new worksheet. (Phase 2.5 Excel Import XLS-1…XLS-5 — the field-schema foundation + the workbook importer/exporter.)
- **Points of Interest, reordered.** The World ▸ **🗺 Points of Interest** sub-tab now lists its tables **Dungeons, then Lairs, then Stashes** (the dungeons-first order most GMs reach for).
- **Tribal domains — clanholds and transitional realms (RR pp.353–354).** A domain now has a **type**, set on its Economy tab in the new **🏷 Domain type** panel: **Ordinary** (the default — unchanged; RAW's word for a non-special domain, and a separate axis from a domain's Civilized/Borderlands/Outlands *classification*), a **Barbarian Clanhold**, or a **Transitional** domain. A *clanhold* is a loose tribal realm — always classified outlands, capped at **125 families per 6-mile hex**, levying **clan warriors only** — its Garrison table (Domains ▸ Military) drops the Conscripts and Militia sections and shows a **⚔ Clan warriors** muster instead: raise up to one warrior per family, arriving already **trained and equipped** as the tribe's customary troop type (no training step), drawing **no wages** (they serve for spoils) and costing the clanhold no morale or revenue, ⅓ of them veterans (RR p.433) — with a **restricted** favors-&-duties set (no councils except for war, no loans, charters, titles, or land grants — only a call to arms), urban settlements held under 12.5% of its peasants (Class VI markets, no urban investment), and a **−2 morale** penalty on any ordinary domain it rules. Its land revenue reflects the 125-family cap — the surplus loses its land contribution, though income *per family* is unchanged. When its chieftain raids or adventures at least once a month (a toggle on the panel), the clanhold's population grows faster (RR p.353). These limits are enforced, not just noted: a clanhold can't be made to take a forbidden favor or duty, its settlements can't be grown past the clanhold ceiling by urban investment (investment is capped at under 250 families / Class VI / 12.5% of peasants, not forbidden), and its vassals' morale reflects its rule. A clanhold can be **decreed Transitional** (the 📜 button on the panel) — the irrevocable middle stage on the road to civilization, where families beyond 125 per hex still pay **half** land value and the realm may field militia and conscripts; after 20 years a transitional domain may be ratified civilized. A domain can also carry a **dominant race** (e.g. *beastman* — which RAW makes a clanhold by default, with a soft advisory for the chaotic-overlord exception). And a **senate cannot sit on a clanhold apex** (RR p.354): the Governance tab says so, and the Senate generator refuses one until the realm is decreed transitional. No house rule needed — the domain type *is* the switch.
- **An Investment tab — all of a domain's investing in one place (RR pp.341, 350–353).** A domain now has its own **Investment** tab (between Demographics and Economy) gathering the three ways a ruler grows his realm: **⚒ Agricultural improvements** (raise a hex's land value over time — moved here from Demographics; press **⚒ Fund improvements** to open a dialog and allocate gp per hex toward its land value, built over time at ~500gp/day with a **⏳ Building now** readout of each hex's dripping budget), a new **🏛 Urban investment** panel, and **Domain Advancement** (the Outlands→Borderlands→Civilized check — moved here from Demographics). The new **Urban investment** panel grows a settlement by investing in its roads, aqueducts, sewers, marketplaces, and walls — **built over time at 500gp/day** (RR p.353). Press **🏗 Order investment** to open a dialog where you enter gp per settlement; the amount is added to that settlement's committed budget and paid out of the treasury at 500gp/day as the Day Clock advances, raising its total investment (and so its maximum-population cap, RR p.350); and the new residents **follow the build** — **1d10 urban families move in for every 1,000gp actually paid** (RR p.351), arriving gradually as the infrastructure goes up rather than all at once. A **⏳ Building now** list shows each in-flight investment's remaining budget, gp paid so far, and days left, with a **⏩ Complete now** button to pay a settlement's whole budget at once when you'd rather not wait. The panel also shows each settlement's families, current cap, market class, and total investment, the families an order will draw, and an advisory **monthly-revenue cap** ("a ruler cannot spend more than his domain's revenue on urban investment each month") — over-spending the cap is flagged but left to GM judgment. (RAW lets the Judge pay investment immediately *or* "deduct the expense at a rate of 500gp per day" — the tool, which exists to do exactly the bookkeeping RAW calls "more than it's worth," uses the 500gp/day build as its model. This is distinct from the Agricultural land-value panel above it, also a build-over-time construction project — RR p.341/p.174; and investments are now run entirely from this Investment tab — the **🏰 Monthly Turn ▸ Construction** sub-tab shows only a read-only cross-domain overview of what is building, with **→ tweak** links back here to adjust it.) The Urban investment panel is also where you **🏘 found a new settlement** — pick an unsettled hex, name it, and seat its first families (RR p.353) — relocated here from the Demographics tab, on the reasoning that founding a town *is* an urban investment. With it gone from Demographics, a hex's settlement **name** and a domain's **urban families** are now **read-only** there (urban population derives from the settlements themselves — you grow it by founding and investing here, not by editing the figure).
- **Terrain Transformation moved to the Demographics tab.** The **🏔 Terrain transformation** panel (how a hex's land changes as its population grows past 186 then 326 families — JJ p.412) now sits on the **Demographics** tab, directly under the Hexes box, where it reads alongside the hex roster it acts on (it was previously on the Economy tab).
- **Garrisons hold abstractly; troops move by muster (RR p.434).** Units no longer carry a home hex or a map pin. A unit sitting in its domain is simply **Garrisoning** — an abstract post with no coordinate — and a unit occupies a hex only when it is with a field army on campaign or standing detached in the field. To send troops anywhere (call them up to an army, throw a sortie against an incursion, relocate them) you **muster** them: the unit assembles and takes up its post after about a week — the realm time period (RR p.434) — shown as **"mustering to …"** with a countdown, instead of marching hex-by-hex. When an army disbands its units fall straight back to **Garrisoning** their owner domain, no return march. The per-unit "Home" picker and the free "March" verb are gone (a lone unit now musters to a hex); the garrison table's Deployment column reads Garrisoning / mustering to … / in an army / Levying / Stood down. A levy in the making now reads **"Levying"** rather than "mustering", so it is never confused with a muster to the field. The unit's domain link is the relational **ownerDomainId** (renamed from the old homeDomainId; the geographic homeHexId + the idle map-hint stationedAtHexId are removed — old saves migrate on load). (Phase 3 Military — the garrison/muster model.)
- **Mustering an army — a captain leads one host, and the name follows the leader (RR p.435).** The 🎖 Muster an Army dialog no longer offers a commander who is **already leading another army** (a captain commands one field force at a time) — busy leaders drop out of the Commander list, so you pick a free character. And the army's name now **keeps step with the commander**: switch who leads and the default *"‹leader›'s Army"* re-derives to match the new leader — **unless you've typed your own name**, which is then left untouched. (Phase 3 Military W1 — muster-dialog polish.)
- **A loyal populace resists spies and thieves (RR p.351).** A hijink pulled off inside one of your domains now answers to that domain's **morale**: against a happy, ruler-loyal populace, a covert operation — spying, stealing, smuggling, assassination, arson, sabotage, racketeering, kidnapping, soliciting — gets **harder** (−1 to −4 on the secret throw as the domain's morale climbs from +1 to +4), the watchful townsfolk on guard for trouble. The penalty shows as a named line in the throw's breakdown ("‹domain› populace") and is locked in when the hijink launches, so it stands even if morale later shifts. A neutral or rebellious domain imposes nothing, and the two hijinks that aren't operations against the local order — **carousing** (overhearing rumors as a tavern patron) and **treasure-hunting** (a dungeon expedition away from the settlement) — are exempt. (Domain morale effects / DC-3 — the loyal-populace term of the RR pp.350–351 domain-morale table, wired into the Hijinks throw.)
- **A small market caps a hijink's take (RR p.359).** A hijink's reward and its target both scale with the perpetrator's level — but the **market class** of the settlement the hijink operates in now caps that *effective* level, the way the rulebook intends: a great city sustains a master-level operation, a backwater village cannot. Class I supports up to a 14th-level operation, Class II 11th, Class III 9th, Class IV 7th, Class V 5th, and a Class VI village or hamlet only 3rd — so a 9th-level thief stealing in a small village earns as a 3rd-level operation (900gp, not 2,700gp), and a 9th-level assassin there can only reach a 1st-to-5th-level target rather than 7th-to-11th. The **throw itself still uses the perpetrator's full class level** (the cap limits the take, never the chance of success), and a hijink with no urban market behind it — the freelance or off-map operator — is uncapped. (Hideout Size, Cost, and Level / RR p.359 — the market-class effective-level cap, wired into the Hijinks reward + target.)
- **Act on a character right from the roster — the selected-character action box (Roster ▸ Characters).** Click any character in the 👥 Characters table and a box above the table fills in with everything you need to act for them: **where they are** (their hex and realm, with a 🏘 badge naming the settlement when they're in one), their **day-activity budget** (the dedicated/ancillary pips, what's still free, what's already been done today, and a 😴 rest warning when they're worn out), and the **full set of actions** they can take — Market, Recruit, Travel, Forage / Hunt, Search, Hijinks, Ventures, Banking, Magic Items, Gladiators, and Spell Research — each **greyed out when the spot they're standing in doesn't allow it** (no market to hand, no hex underfoot, the Gladiators rule switched off), with the reason on hover. An enabled action does the thing or jumps to where you do it, with the character already filled in — and an action the character is **already engaged in** (a journey under way, a hiring drive in progress, an active venture, a hijink, the arena) shows as a filled dark button that jumps straight to the thing in progress instead of starting a new one. A character already travelling can't begin a second journey — the Travel action opens the journey they're on (everywhere, not just here). This **replaces** the per-character Activities ▸ Activities dashboard table — that table has been removed from Activities (its *Current Activities* reject / roll-back surface stays) — bringing those actions to where you pick the character. A row click (or the name) selects the character for the box; the full sheet stays one click away on the row's **edit** link or the box's **✎ Edit sheet** button. The box also surfaces the character's **henchman status** — whom they serve (⚔ henchman / 🛠 specialist / 🛡 follower / 🤝 hired by their patron) or, for a lord, how many **henchmen they command** against their cap — and, when they belong to a **party**, a link straight to it (👥 Roster ▸ Parties). With those moved into the box, the 👥 Characters **table is leaner**: the **Activity** column is gone (the box's day-budget already shows what each character is doing), and **Roles** is renamed **Role** and kept to a single line — the ruler, magistrate, senator (🏛 a seat in a realm's senate), and gladiator-of-a-school posts (the retainer and party tags now live in the box).
- **Magic research counts as a day's work — and it's on the Activities dashboard now (RR p.388).** An in-progress research project **dedicates the researcher's day** (and each named assistant's), so a researching mage now shows as busy on the 📋 **Activities** dashboard and in **Current Activities**, fills a dedicated slot, and — like any dedicated task — caps what else they can do that day (you can't both research and administer a domain). Research also advances **day-by-day on the Day Clock** now (RR p.388): each day an in-progress project gains its researcher + assistants' rate, completing after the rule's number of days — advance the Day Clock to research, or commit the month and it ticks the days for you (no more waiting for the monthly turn to lump it in). The character-sheet **⚗ Research** card is clearer too: in place of the confusing "N day(s) of research remaining … advances each monthly turn", it now reads "N day(s) of dedicated work left. Advance the Day Clock (or commit the month) to research." And the stale **"Spell Research · coming"** placeholders are gone: the Activities sub-tab and its Phase-4.6 "coming soon" panel are removed (research has shipped on the character sheet since the Arcane Domain landed), and the Roster ▸ Characters **Spell Research** action now opens an arcane caster's ⚗ Research tab (greyed for non-casters) instead of reading "Coming soon". (Activity Budget AB-4 + the SR-1 per-day-accrual fix — derive-don't-store, no new house rule / event / entity / prefix.)

- **Armies march, scout, and conquer — the campaign layer arrives.** The merged **Characters ▸ Parties** view — now home to your parties, armies, and loose units together — is where war is waged across the map (RR pp.447–460). By default the rulebook's own **Supply Simplified** governs — just pay the weekly cost, and only trace the line when the army is cut off, crossing hostile or unsettled country, or more than a day's march from a road or waterway. Fall out of supply and the troops go **underfed, then starving, then — in the waterless desert — parched**, every unsupplied week a calamity that tests their loyalty. **Severe weather now bites the campaign by the book (RR p.449):** sweltering heat raises the supply cost a quarter (the troops drink more) and doubles the penalties of going unsupplied, foul weather breeds disease — frigid or cold exposure, rain or snow — giving the army a weekly chance an epidemic breaks out in the ranks (each unit saving against it; you resolve the saves), and rain or snow **hampers the scouts** — a −2 penalty to the army's reconnaissance throw, or −4 in barrens and desert where wind and storms whip the sand and snow into a blinding screen — so a marching column has a harder time fixing the enemy in the murk; the supply check and the disease check are both laid out in the day's review, and the reconnaissance penalty shows on the recon record itself. A cut-off army can **feed off the land** — *requisition* grain from a domain's peasants (35gp a family, once a year, and they survive it) or *loot* it outright (at the cost of one family per 15gp torn away). The army card carries a **Supply** panel — its status, weekly cost, base value, and line state; the Simplified toggle; the supply bases you designate — friendly or **occupied** domains, a **captured stronghold**, or a **border fort the army raises in the field** (a 10,000gp construction that drops a Class VI forward base and relays the supply line back to a distant capital, RR p.451); and the Requisition / Loot buttons when it's gone hungry — and an army of 1,200+ is its own travelling market for equipment, until its lines are cut — and that market is now **shoppable**: a **🛒 Shop in camp** button on the army card opens the Trade Wizard seated on the baggage train, so the host buys and sells equipment in the field at its own market class (RR p.452), the camp's class recorded on each transaction.
- **Strongholds fall — the siege engine arrives (RR pp.473–485).** Send an army against a garrisoned stronghold and lay siege in **📋 Review ▸ 🏯 Sieges**. By default a siege resolves the rulebook's **Sieges Simplified** way — the tool weighs the stronghold's structural hit points and defensive capacity (read by the book from its actual structures, RR r10 p.132 — a keep's own 16,000 hit points and 6-unit capacity — or estimated from its value when it has none recorded) against your **unit advantage** (your units and siege engines, less the defenders') and tells you how many days it takes to storm, multiplied for an inaccessible site (riverbank, peninsula, island, mountain redoubt) — or warns you're too weak and can only **blockade and starve them out**. Go **detailed** and the methods open up: **blockade** the place (encircle it, cut its supply line, watch its larder dwindle week by week toward starvation), **reduce** its walls with a **siege train** of catapults and trebuchets (each day's bombardment tears down structural hit points and opens **breaches** that let more troops pour in), and when it's softened, **assault** it — which fights as a full battle in the battle engine, the defenders holding the high ground of their walls. The Day Clock runs the clock: when a siege's time runs out, the day's review says the stronghold can be taken and **holds the clock** until you resolve it — capture, raze, accept surrender, or lift the siege and march away.
- **Battles are fought in the tool — the full ACKS II battle engine arrives.** When two forces stand in the same hex — an incursion warband against the domain garrison, a monster band against an army on the march — the new **📋 Review ▸ 🎌 Battles** view fights it by the book (RR pp.461–472). Set up the engagement in the launcher: pick each side (an army, a domain's garrison, or a monster band — bands fight **as themselves**, priced straight off their Monstrous Manual battle ratings, chunked into the rulebook's platoons), each side's **strategic stance** (offensive, defensive, evasive) and the **state of awareness** from reconnaissance, and the tool derives the **strategic situation** exactly as printed — pitched battle, meeting engagement, rear guard action, skirmish, ambush, envelopment, deep envelopment — with who is **surprised**, which divisions may deploy (a meeting engagement is vanguards only; an evader covers with its rear guard), and which zones the surprised side cannot fill. The W2 incursion box offers a one-click **🎌 battle** that prefills all of it — a ruler whose scouts failed arrives *surprised by the monsters in his own march*. The battle panel then walks the rulebook's battle: units deploy across the **Left / Center / Right / Reserve zones** under their commanders' leadership-ability caps (auto-deployed; rearrange freely before the lines close), **missile and loose units** read straight off each troop type's printed kit (loose units soak hits at half value and withdraw disordered instead of dying), a commander's **strategic ability raises his division's BRs** (+½ each at +3, +1 at +5 — the printed captain example reproduces exactly), and each **~10-minute battle turn** rolls the zones' attack throws (17+ missile, 16+ melee; +2 against a broken zone or a surprised foe, −2 against high ground, hits spilling past a broken zone into its neighbours — and severe weather hampers the archery, rain or snow at −2 and a storm at −4 to every missile throw on both sides, RR p.449), then checks **morale at the break point** — one third of the army gone and every unit rolls on the Unit Morale table (Rout / Flee / Waver / Stand Firm / Rally), each result landing before the next unit rolls, so a wavering line can cascade off the field exactly as intended. Reinforcements throw to arrive by terrain; the latest turn can be **rerolled whole**; either side may **withdraw** between turns. **Heroes ride along both ways** (RR pp.466–467): add a character as a **heroic unit** (his BR computed by the printed formula — the rulebook's own worked example lands on the decimal) with qualification checked at the battle's scale, or **declare a heroic foray** — the panel frames the at-table fight for you (the foes drawn from the opposing zone, proportionally down to creature counts exactly as the wolf-rider example does it, arriving in 1–4 groups at the terrain's printed battlefield encounter distances), you resolve it at the table under the normal combat rules, and the recorded outcome strips the staked BR from the enemy army at the right step of the turn; heroes foraying opposite each other meet **heroes-versus-heroes**. **Elite troops** now bite too: with the elite-troops rule on, a flagged unit's share of the throws attacks at +1 (RR p.434). And when the field is decided, the **aftermath** settles it all (RR pp.469–472): the loser retreats (into the stronghold if one stands in the hex — the siege phase will pick that up), eligible victors roll **pursuit** (light cavalry and flyers at 11+, everyone at +4 once the loser's cavalry is gone, a cumulative penalty for every turn an evader bought, a natural 20 always telling), casualties split by the book (a destroyed unit half dead and half wounded — the victor's wounded return in a week, the loser's march off as **prisoners**; routed units bleed dead, deserters, and captives), each fallen **officer rolls Mortal Wounds** — you pick the outcome band (death, capture, escape, or a survivable wound; a routed officer may simply surrender instead) and the panel **resolves and applies it through the rulebook's Mortal Wounds rules**, so a wounded officer comes away with a real permanent wound, the standing loyalty penalty it carries, and a bed-rest convalescence the Day Clock heals — while a killed officer is recorded dead (an assault on a stronghold wounds the defenders' officers the same way, since a siege is fought as a battle), the **spoils** count up (a month's wages of every unit destroyed or routed, plus 40gp a prisoner; a monster band's spoils are its own treasure), and the **experience** flows as printed — the leader and his division commanders split the value of the units defeated, every winning soldier earns toward the militia-to-mercenary and mercenary-to-fighter promotions, and one click **applies it all to the world**: real casualties on the units and bands that fought, XP on the commanders, the battle filed in the Event Log with its full turn-by-turn record. Battle turns stamp the campaign's sub-day context, so every battle event knows exactly which ten minutes of which day it belongs to.
- **Domains now suffer incursions — the Vagaries of Incursion arrive (optional).** Turn on **Vagaries of Incursion** (⚙ House Rules ▸ ⚔ Encounters; the rulebook calls the chapter strictly optional, so it's off by default — the bundled demo ships with it on) and every domain rolls the **Daily Domain Encounter Probability** (JJ p.101) as the Day Clock advances: each domain's chance is derived live from its real geography — its territory in hexes, its **dangerous borders** (a domain pressed against the unsettled wild counts as far larger prey, JJ p.102: the tool reads the hex map — a face is safe where a neighbouring domain, open water, or a river edge covers it — and judges the *isolated / spearhead / flank / line* configuration for you, with an override field when your judgment differs), and the **state of its defences** (a domain whose garrison or stronghold is under the required strength reads one classification worse — a bankrupt outlands march rolls as unsettled wilderness, exactly like the rulebook's worked example). The domain's **Overview** tab gains a **⚔ Domain encounters** box (just beneath the Morale block) showing the whole derivation. When the dice strike, the day's review lays out the entire arrival as one record: **who comes** (rolled on the hex's own terrain encounter table, graded by the territory's rarity), whether they **linger or migrate** (their Lair % — lingerers may arrive at full lair strength, with treasure), their **domain reaction** (the JJ p.103 table, 2d6 swung by the domain's morale and the clash of alignments — doubled when the arrivals outmatch the garrison; beasts and mindless things never come to trade), **reconnaissance for both sides** (RR p.452 — a ruler whose scouts fail may be entirely *unaware* monsters have settled in his lands, and the record says so), and the **platoon-scale Battle Rating comparison** against the domain's actual garrison with the rulebook's trigger lines spelled out (assault the stronghold, pitched battle, driven off, loot and depart — JJ pp.104–106). Commit the day and the arrivals are **real**: a band placed on the map (visible in the domain panel, the 🐉 Monsters Groups table, and the Event Log with its full verdict), migrating bands wander on across your world with the Day Clock, lingering bands hold their ground as a standing threat — and a lingering band that came in peace settles a new (undiscovered) lair on the spot. Meeting an incursion band on the road runs the full encounter panel, which knows the band's attitude toward the domain and carries it into the parley (JJ p.104). A wandering band that physically crosses a border already counts as that domain's encounter for the day — the dice are never rolled twice. And peasants notice: a neutral band the ruler leaves unchallenged costs −1 on the domain's next morale roll (JJ p.103). And when it comes to blows, the battle itself is one click away — garrison versus warband at platoon scale, fought in the battle engine (above); the rulebook's full **Mass Combat for Domain Encounters** tables (the whole Monstrous Manual pre-organized into platoons, JJ pp.106–109) are built in, pricing every band for that day.
- **Your soldiers are now real units — and armies exist.** The warfare layer's foundation (ACKS II RR pp.427–446) arrives: every garrison unit and mercenary company detachment is now a first-class **Unit** in the campaign — browse, inspect, and edit them all under **🔍 Inspector ▸ 🪖 Unit** — carrying its troop type, race, and equipment loadout, veteran and elite status, its recruitment source (mercenary, conscript, militia…), casualties, unit loyalty and the calamities that test it (routing, heavy losses, hunger, an empty pay chest — RR p.430), its supply condition, and **where it is stationed** (a domain's garrison, a captain's company, an army, a hex, a fortification). Existing campaigns lift their garrisons automatically — nothing moves, nothing renames, the domain panel's Military tab works exactly as before. **Armies** join them: a leader, embedded **divisions** with commanders and adjutants, a strategic stance, and supply bases — with the rulebook's organization rules checking the muster (at least three units; divisions capped by the leader's leadership ability; commanders qualified for the army's scale — RR pp.435–437) and the leader's **officer characteristics** (leadership ability, strategic ability, morale modifier — including an adjutant's counsel and its cost) derived from his sheet exactly as printed. You raise one **in the fiction** — a **🎖 Muster army** button on a character's sheet (he becomes its commander) and on a domain's Military tab (its ruler leads, his garrison pre-selected), by **turning an existing party into an army** (a 🎖 Muster button on the party — its members become the army's officers and their mercenary companies its first units, and the party is dissolved), or **🎖 Muster a new army** on the merged **Characters ▸ Parties** view: pick the commander, name the host, and check off the troops to station, with the muster's troop total, battle rating, the commander's leadership, and any organization warnings shown live before you commit. That view now gathers **parties and armies together in click-to-open tables** (the old World ▸ Armies tab is gone — its detail card moved here): an army shows its **officers** exactly as a party shows its members, and lists the **units travelling with it** with their load-pace ("encumbrance" — the slowest unit, the army's bottleneck, flagged) and supply mode. And when an army (or a unit marching to muster) is on the move, its **travel panel shows the same information as a party's journey, adapted to the force** — its units and officers in place of party members, each unit's march pace, and its **supplies** (RR p.450) in place of a party's personal rations and water — and you steer its march (destination and waypoints) from the map exactly as you do a party's journey. And you can **start a battle straight from the army** — a 🎌 *Start a battle* button sets it as side A and the launcher offers the forces standing on its hex (a rival army, a domain's garrison, a monster band) as the foe to pick in one click. Troops don't teleport: units **at the muster point** fall in at once, while troops **elsewhere are called up** and **march to the muster point** — each shown with how far off it is (hexes, miles, and days at its own pace), arriving and joining the army as the Day Clock advances. You can call up more reinforcements to a standing army the same way (the army card's **＋ Call up units**). A mustered army you no longer need can be **disbanded**, its units returning to the pool to re-muster later; and 🔍 Inspector ▸ 🎖 Army remains the free-form authoring path. You can also **add a unit to — or remove it from — a field army right on a domain's Garrison Units table**: any garrison unit (mercenary, conscript, or militia) standing where a field army is mustered shows a **🎖 Add to army** button (with a chooser when several armies are present there), and a unit already with an army shows **🏸 Leave army**, which detaches it where the army stands (a unit still marching in instead has its call-up cancelled and returns home). And a free garrison unit can **march anywhere on its own** — a **🚶 March** button sets up a journey at unit scale (pick a destination, choose it on the map, and set the pace, just like starting a party's journey, but with no supply line — a lone unit carries what it carries): the unit leaves its garrison and marches there hex by hex as the Day Clock advances, halting at the destination, with a **🛑 Stop march** to call it off where it stands. Characters who lead armies, command divisions, or lieutenant a unit show it on the Activities dashboard.
- **Deploy the garrison to meet a domain threat (JJ pp.104–106).** When an incursion band stands in your domain, you can now sally a force out to meet it in the field — "not every encounter requires the ruler to sally forth… that's what garrisons are for!" A domain's **Military tab gains an ⚔ Active Threats table** (and the State-of-the-Domain incursion box an **⚔ Deploy garrison** button) listing each live band with **whether the ruler is aware of it** (a dedicated *Aware?* column — ✓ Aware in green, ⚠ Unaware in red), its attitude, disposition, and a **platoon-scale Battle Rating forecast** — your garrison versus the band, and what it means (*they fight* / *driven off*) — so a threat you didn't answer when it first appeared is dealt with here later. **But you can only sally against a threat the ruler has detected**: a band whose arrival slipped past the domain's reconnaissance (an *UNAWARE* ruler — JJ p.103, RR p.452) shows the **⚔ Deploy garrison** and **🎌 battle** affordances greyed, with a tooltip saying why, until your scouts locate it — you can't march the garrison after something it hasn't found (the engine refuses the deploy with reason `ruler-unaware`). The general ⚔ New-battle launcher stays open as the Judge's override. **⚔ Deploy garrison** opens a sally muster: choose the units (the domain's garrison is pre-checked; troops elsewhere are called up and march in), a commander (the ruler may lead in person), and a **rally point** (the domain's seat by default — JJ p.103), with the **force-versus-band BR and the predicted outcome shown live** as you toggle the force — and, just as the army card does after a muster, an **army-organization advisory** (RR pp.435–437) warning when the force you've assembled is sub-strength (fewer than three units, no commander, a commander outmatched by its scale), so you see it before you commit rather than after. The chosen units muster as a temporary army and **march to the band** on the same travel engine as any army; when the force arrives, the meeting **resolves by the book** — a weak Unfriendly band is **driven off** (repelled from the domain), while a Hostile or strong-Unfriendly band gives **battle** in the battle engine (deploying against a Neutral, Mercantilist, or Friendly band turns it Unfriendly first — JJ p.104). The army's card shows its **reaction status** — marching to meet them, in position, or in battle — and a band that **wanders off before you arrive is now chased automatically**: each day the marching force re-routes to follow the band wherever it has gone (a faster force closes the gap and catches it), so a migrating warband can no longer simply outwalk a sally you've launched. The **🧭 Re-route to the band** button stays as a manual override (re-route at once, or send the force to intercept ahead), and **🏠 Recall** sends the force marching home once the work is done. Units now keep a **home garrison** they return to: disbanding an army — or recalling a sally — sends each unit **marching back to its home hex**, never teleporting (set or change a unit's Home on the domain's Military tab), and that Military tab now shows each unit's current **Location** — its post in the domain, or away with a field army / marching to muster — and gathers the domain's **Active Field Armies** (the host mustered from it, the reaction sallies now afield) in their own table, each row opening the army in Characters ▸ Parties.

Pick the adventurers, the foe (an active lair, or a monster type + count), and any army that marches with them; the tool computes the **expedition level**, the encounter's **challenge** (its monster level and how dangerous this particular band is by its experience value), how far your platoon-scale **army** reduces that danger (its Battle Rating, halved for each level the monsters outclass it, the contribution capped at +2 — the rulebook's worked examples reproduce exactly), and the **scaling** that makes a large expedition bleed more than a small party, then rolls the result from Catastrophic (the expedition slain) to Stupendous (no losses, double treasure). Casualties are applied through the rulebook's **Mortal Wounds** — each fallen adventurer takes a real wound or death, his survivors carrying on — participating **units lose half their strength**, and a lair's treasure (≈ 4× its experience value, scaled by how well the fight went) plus the combat experience flow to the survivors through the normal adventure-result, with magic-item rolls by monster level. And **clearing a lair this way moves the hex toward secure** (JJ p.68) — the wilderness-clearing loop. Propose → review → ⟳ reroll → commit, the same as every other foray.
- **Characters grow old (RR p.19).** Set a character's age on their sheet — or 🎲 roll a starting age for their class — and they age as the months turn. Crossing from Adult into Middle Aged, then Old, then Ancient adjusts their attributes by the rulebook's table (the veteran loses Strength, Dexterity, and Constitution, never to be regained), and reaching old age brings the **death-from-old-age** saving throw, due within 1d12 months of passing each threshold — racial Old age + Constitution, Ancient age + Constitution, and the racial maximum (and each year thereafter) — with the falling Constitution lowering the very thresholds, just as the rulebook's worked example shows. The Identity tab shows the age, life stage, and the next old-age save the character faces; the monthly turn previews each aging change for you to confirm before it lands, and records milestones and old-age deaths in the Event Log. Elves and Nobirans are ageless; a character whose age you leave unset simply doesn't age — so existing campaigns are untouched until you choose to track it.
- **Characters can transform into monsters (JJ pp.94–95).** A character's **Stats** tab gains a **Transformations** section: 🐺 **Transform** them into another form — a werewolf, an undead thing, whatever a curse or ritual made of them — and they make the rulebook's **Spells save** to keep their class abilities, then begin to **drift** toward the new form's alignment, an alignment-drift Death save falling due every so often (succeed and they keep their own mind; fail and they think as the beast now). A lycanthrope who **rejects the gift**, or an undead **After the Flesh**, never drifts. The drift clock ticks on the monthly turn, you can roll a drift save by hand, and a reversible transformation can be **reverted** to the original self. The *cause* — the spell, the bite, the ritual — arrives with the Magic layer; this lands the ledger and the clock it rides.
- **Levy and train your realm's soldiers — conscripts & militia on the Domain Military tab (RR pp.430–433).** A domain's **Military tab** presents one unified **Garrison** table — 🪖 Mercenaries, ⚒ Conscripts, and 🛡 Militia in three sections sharing the same columns (Name · Type · Count · Wage · Sub · BR · Morale · Home · Deployment) and footed by a single **— full garrison** total row that sums all three sections (mercenaries + conscripts + militia). The table lists **every one of the realm's military units — those posted in the garrison and those away with a field army alike** — a unit mustered or marching to an army still appears, its **Deployment** column naming the field army and where it stands (a unit at its post reads its garrison hex); a unit in the field is managed through its army (Characters ▸ Parties), so its in-table Split / Merge / remove actions are held. Every unit's name opens its **sheet** where its details are edited (a unit's *type* is set by recruiting or training, never by hand — the 🔍 Inspector is the raw path). You **levy** conscripts and militia from the realm through a dialog that asks how many and which garrison hex to home them at (capped at the rulebook's one-in-ten and two-in-ten of families; blocked when the realm is too unhappy) — and **levying now takes time** (RR p.430): the levied troops arrive in stages, ½ the first week, ¼ the second, the rest the third, showing as *⏳ mustering — N of M arrived (D days left)* in the Deployment column and assembling into the garrison as the Day Clock advances (an Advance-Month musters them too); a still-mustering levy can't yet be trained, marched, or merged, and its full commit reserves the levy cap from the moment you raise it. You **train** the assembled levies into professional troops through a **🎓 Train dialog** that lays out the RR p.431 procedure — the realm's **total levy budget** up front (the whole pool the per-type numbers divide: how many of this source you've levied, trained versus raw recruits, the treasury you can spend on training, and how many more you can still raise) and then the per-type **qualifying limits** (only half of *all* the conscripts/militia you levied can become heavy infantry, an eighth heavy cavalry, …), the training time and cost, and the trained unit's stats — and **enforces the cap pool-wide**: the limit is a fraction of the **whole levied pool**, not the one unit being trained, so splitting a levy into smaller units before training can't dodge it (RR p.431/432) — you pick how many to train (up to what the pool still allows, after any already trained) and the **unqualified remainder splits off as an untrained levy** — and **training now takes time** (RR p.431): the cost is paid up front, then the recruits *drill* for the troop type's months (a month for light infantry, up to a year for cataphracts) before they're ready, showing as *🎓 training as … (N days left)* in the Deployment column — raw levies still, who can't fight, march, be re-trained, or merge — and they **join the garrison as the trained type when the Day Clock carries them through the training period** (an Advance-Month carries them too; the drilling cohort still counts against the type's pool cap meanwhile, so two trainings of one type can't overrun it). You can also **divide** a levy into units (Split/Merge — e.g. to train half as heavy infantry and half as light), and **release** them from service (untrained return to the farms; trained conscripts become mercenaries or brigands). Each table shows the **available pool** — the cap minus *everyone ever raised*, so **casualties are sticky**: the fallen hold their family slot, and you can't instantly re-levy the dead (only population growth or replenishment refills, exactly as the rulebook's Marcus example) — and casualties **recover over time** at 5% of the cap per year, each monthly turn. **Militia** carry their full RAW cost while called up — each costs a family of revenue and −1/−2 domain morale — until you **🏠 send them home**, which *stands them down to the rolls* rather than disbanding them: they stay listed (greyed, Deployment reads *at home*), draw **no wages** while at home (RR p.432), and can be **called up** again without re-levying. A trained-and-equipped militia still counts toward the garrison even at rest (RR p.341). To return militia to the population (and free the levy slot) you **release** them instead. Training militia carries the rulebook's risk, too: should the domain fall to **Turbulent** morale or worse, those armed, trained men are the first drawn into the resulting banditry (RR p.433) — *Domain banditry takes the field* then fields them as their trained troop type, not peasant rabble. And when you **🎖 muster a field army** — now launched from the domain's **Active Field Armies** card — its picker lists the realm's conscript and militia units (clearly labeled) to call up — a militia called into the field resumes its revenue/morale cost on the domain. (The old army "Troop depth" levy panel is gone: levying is a domain act, and an army now shows only its read-only troop composition.) And the **Favors & Duties** that were once just promissory notes now bite: a **Call to Arms** duty musters a vassal's levy into the liege's host, and the **Troops** favor stations soldiers under a vassal at the lord's expense — revoke either and they disband. Beyond your own peasants, you can now **recruit mercenary units at realm scale** (RR p.428): the 🪖 Mercenaries section of the Garrison table gains a **🪖 Recruit (realm)** button → a dialog that hires whole professional, equipped units at your realm's tier (derived from its family count — barony through continent), capped by the tier's per-period availability, for a recruitment fee paid from the treasury — and, like a levy, the troops **arrive ½/¼/remainder over the tier's time period** (a week for a barony or county, a month for a duchy, a season for a kingdom), mustering into the garrison as the Day Clock advances. Unlike a levy, hiring foreign mercenaries isn't blocked by unrest — a rebellious realm can still recruit. Alongside the troops you can **recruit the realm's officers and military specialists** (RR p.428) — artillerists, armorers, creature handlers, marshals, and mercenary officers from lieutenant to general — through a **🎖 Recruit (officer/specialist)** button beside it: each is an individual, hired either as a **lightweight stub** (a named NPC with its type, level, and wage — you flesh out the rest later) or as a **full character rolled on the spot**, and a lightweight NPC is never a dead end — its sheet carries an **⬆ Expand to full** button that rolls it up to a complete character whenever you like (mercenary officers arrive with their rulebook leadership, strategic ability, and command proficiencies — RR p.171). (This is the army-scale complement to the settlement-scale **+ Recruit unit** Recruiting Wizard, which solicits individual hirelings at a market.) And the Military tab now opens with a **🏰 Realm forces** summary — the rulebook's *Vassal Troops by Realm Size* table (RR p.434) read for your realm: its tier (from its family count — this domain plus its sub-vassals), the **maximum standing army** it can field, the **realm-troops wage budget** and garrison baseline that tier implies, and the troops it currently fields against them — flagged in red when you've raised more than the realm's nominal capacity (a frontier march or a realm at total war runs higher, as the rulebook notes).

### Fixed
- **Henchman rulers now earn full domain XP (RR p.342).** A character ruling a domain as a vassal-henchman was having the experience from that domain's income **halved** — on top of the rulebook-correct subtraction of their wage from the XP basis. But domain income is the one place the rules say *not* to take the henchman half-share: "Henchmen vassals subtract their expected monthly wage… however, they do not reduce earned XP from domains by 50%" (RR p.342). The extra ½ is gone, so a henchman ruler earns the same domain XP a player-character ruler would from the same net income (the wage subtraction still stands). The half-share is unchanged where the rulebook *does* call for it — adventuring, construction, and mercantile ventures.
- **A domain's passive-investment income no longer vanishes when you reload the campaign.** A passive investment (a distillery, a trade share, and the like) paying its monthly return into a domain treasury credited the treasury *directly*, bypassing the treasury's stash — so on the next load the routine "reconcile the treasury against its stash" pass overwrote the credit back down, and the month's payout was silently lost (observed: a 300 gp distillery payout that read correctly in-session reverted on reload). The payout now goes through the same canonical treasury setter every other monthly money-move uses, so it lands in the stash and survives a save/reload.
- A domain tagged or named **"empire"** is now correctly treated as an **emperor's** realm (it had fallen through to baron) — so realm-scale calculations that key off the title (Favors & Duties, the demographic command structure, the new war vagaries' unit sizes) read an empire at the right tier.
- **Assassins and bards now get the correct saving throws.** Both advance on the *fighter* saving-throw progression (RR: "Assassin (Fighter)…", "Bard (Fighter)…"), but the engine was giving them the *thief* progression — so a standard assassin or bard had the wrong Paralysis/Death/Blast/Implements/Spells targets. Corrected. (The four base class save tables themselves — fighter / mage / cleric / thief — were verified against RAW and are unchanged; a new test now pins all four.)
- **The stronghold structure costs now match the rulebook (RR r10 p.132).** The structure catalog behind the *stronghold composed of buildings* house rule had drifted from the current rulebook — a stone keep read 75,000 gp (it's **125,000**), a barbican 38,000 (it's **20,000**), the towers ran nearly double, a wooden building 1,500 (it's **350**), the rampart 2,500 (it's **300**) — and it was missing structures the rulebook lists (the small square tower, both gatehouse sizes, barracks, stable-barracks, granary, hospital, headquarters, the crude/cheap variants, and the wall-walk). The whole table was rebuilt to RR r10 p.132 — all 33 structures at their correct costs, and each now also carries its **structural hit points, armor class, and unit capacity** from the rulebook (read directly by the siege engine — see *Strongholds fall* above). For most campaigns nothing changes — the rule is off by default, so a domain's stronghold value comes from its recorded build value, untouched — but a campaign that builds strongholds structure-by-structure now prices them by the book.

## [0.24.0] - 2026-06-12

### Added
- **Monster lairs are now first-class places in your world.** A monster's lair — its den, warren, or webbed deep — is tracked as a proper entity bound to a hex, holding its inhabitants (the monster group that dens there), its treasure, its discovery state, and its own history, with a lifecycle (active → cleared / abandoned / destroyed). A new **Lair Wizard** (🔍 Inspector ▸ Create ▸ Lair) lets you **author a lair** by hand — placing it on a hex now, or holding it in reserve to drop into a hex later — or **seed a wilderness hex**, which rolls how many lairs that hex's terrain supports (JJ p.69, *Lairs per Hex* — from a sparse steppe to a teeming jungle) and lays down that many undetailed lairs for you to flesh out as your party explores. Only unsettled hexes seed; a hex inside a domain holds none. Clearing a lair through an adventure result marks it **cleared** (the structure remains, ready to be reoccupied) rather than erasing it — and clearing or destroying a lair fells the monster band that held it (abandoning it sends them away alive), so the hex's monsters always agree with the lair's state. The bundled demo ships the three lairs that beset the March of Saltspur — the Grey Pack's den, the Drowned Causeway of the Mere-Folk, and the Weaver Deeps — each tied to the monster band that holds it; the Frontier Barony template's goblin warren is likewise now a real lair you can inspect. The Lair Wizard can now **generate a lair from a built-in monster catalog** — pick a monster (orc, dire wolf, giant spider, and ~280 more, each with its ACKS stats from the Monstrous Manual) and the wizard fills in its lair chance and treasure type and rolls how many den there into a monster band bound to the lair; type a monster the catalog doesn't have and you author it freehand as before. And when a travelling party enters a hex that holds an active lair, the journey's encounter check now **surfaces that lair** instead of a generic "something happens" — and rolls the monster's **Lair %** to decide what you actually meet: the lair itself, or a **wandering band out from it** (no hoard with them, and their home stays unlocated — a tracker can follow the spoor back; see below). An encounter in a hex you seeded proposes one of *its* unauthored lairs to flesh out, so the wilderness remembers what dens where.
- **Your party can now search a hex for what hides in it.** A **🔍 Search hex** button on any hex card — and a **🔍 Search** quick-action beside Market / Recruit / Travel / Forage on the Activities dashboard (greyed, with the reason, for a character with no location set) — runs the ACKS wilderness search (RR pp.276–277): each search-hour is one short task of the searcher's day, thrown against a target set by the party's expedition speed (a fast, unencumbered party covers more ground; rough terrain slows the sweep), +4 if anyone has **Tracking**, −4 when hunting a *specific* site; the Judge rolls secretly, so a failure tells the players nothing — and each hour of rummaging risks its own wandering encounter (it is entirely possible to fail the search and *stumble onto the lair anyway* — and when the hex holds no lair to stumble onto, the hour's meeting is named from the same terrain tables as any wandering encounter). A success uncovers one of the hex's undiscovered lairs (a well-hidden lair can demand a better throw); a companion with **Land Surveying** can assess *how many* points of interest the hex holds (and a botched assessment confidently reports a wrong number). The latest search throw has a **one-click re-roll** — it re-throws just that die (the hour's encounter and survey stand) and takes no extra time, swapping the find or the miss to match. Found lairs are narrated to the campaign chronicle; fruitless hours are not. Wherever monsters turn up away from their lair — a journey's day-log row, the encounter panel itself (so a band met at your **night camp** can be tracked too), or a meeting risked during a search-hour — a **🐾 Track** button picks up their trail (the full tracking rules are their own entry below); once a band's den is found, the button gives way to a plain **→ lair** link. (A band met *while searching* gives you spoor, not the den — only the search throw, or following the trail, uncovers it.) And a hex's live lairs now **flag the securing problem** (RR p.338): the hex card warns when lairs stand in the way of settling the hex.
- **A Monsters view — the lairs, and the bands abroad.** **World ▸ 🐉 Monsters** lists every lair in the campaign — monster, hex, status, whether the players know of it, inhabitants, treasure — with filters and search; and **beneath the lairs, a Groups table tracks the bands abroad** — the monsters *not* settled in a lair: a population that abandoned (or outlived) its den roams as a **migrant** (named, counted, and located — or wandering with no fixed hex, the table says which den it came out of — and it genuinely **wanders the map** with the Day Clock; see *Monsters now move in your world*), a pack on a fleeing party's trail shows as a **pursuer** with its quarry and the gap in miles (one click opens the chase; an undecided pursuit reads *"pursuit offered — decide"*), a band a party is following shows as **tracked**, with whoever is on its trail (one click opens the follow), and a band walking home after a hunt shows as **🏠 homing**, with the den it returns to. The bands on this table are **live in the world**: a wandering encounter that rolls their monster in their hex meets *them*. The tab's badge counts them all. Click any lair (there, on a hex card, or from a journey's day log) for its **detail panel**: the inhabitants (the bound monster band and its losses, named leaders), the treasure type, the discovery history, and one-click lifecycle actions — **⚔ Attack a lair the party has found**: with your characters standing at its hex, one click opens the assault as a full encounter (the den's living population against everyone present — roll the opening distance and surprise, parley a surrender, or take it to combat; the battle itself stays yours to resolve, and recording a *cleared* adventure outcome flips the lair; the den remembers each assault, so the next one recalls how the last went) — or mark it **discovered**, **cleared**, **abandoned**, or **destroyed**, **populate** an unauthored lair from the monster catalog on the spot, or **reveal** a held-in-reserve lair into a hex of your choosing (its monsters move in with it). The hex card's old free-text lair list is replaced by the real thing — and every hex now states its **maximum lairs** (JJ p.69: the full dice spread in the unsettled wild; a third, half, or two-thirds of that in civilized, borderlands, or outlands territory) on the hex card (red when the hex is full), in the World ▸ Hexes lair tooltips (whose counts now read the real lair records), and in the Lair Wizard's seeding preview.
- **Wilderness encounters now run the real ACKS rules — and an encounter is a thing in your world.** When the dice say "something happens," the tool now rolls the rulebook's actual wilderness encounter throw (JJ pp.41–42): a d20 on the territory-classification table — so what turns up depends on **whose land you're in** (a civilized march mostly offers travellers, herds, and patrols; the trackless wild offers monsters) — with **roads safer but not safe** (travel on a road reads a safer column of the same table), night watches riskier, and monster results graded by **rarity** (the deeper the wild, the rarer and nastier — JJ p.44). **The dice then name the creature:** the rulebook’s own terrain tables — the eighteen *Monster Encounter by Terrain Type and Rarity* tables and the *Civilized Encounter by Terrain Type* table (JJ pp.43–62) — roll who it actually is for that hex’s terrain, sub-type and all: wolves on the taiga, rock baboons in the barrens, a patrol on a farmland road, with the printed regional species kept by name (the sled dogs of the tundra, the quelea swarms of the desert rivers); the few table results the catalog leaves to the Judge (a Dragon, a Genie, a fire Elemental) arrive **labelled** for you to detail. **And the land answers** (JJ p.43): the creature’s Lair % decides whether you’ve met it *in its lair* or abroad — in-lair, it IS the hex’s known den when one of that kind is there, or it **details one of the hex’s seeded, unauthored lairs**, or **reveals a lair you held in reserve**, or **founds a new den on the spot** (known to the party — they are looking at it); the same monster met abroad near its den is a **band out from it** (no hoard with them, the den unlocated) — and a band **already abroad in the hex** (a pack mid-hunt, a roaming remnant from the Monsters tab's Groups table) answers the roll **as itself**: the meeting names who they are and what they're doing, never a second anonymous band. The encounter panel shows the table roll and the lair verdict in full arithmetic, and until the meeting is truly joined you can **reroll on the same table or pick from it by hand** (switching the rarity column if you like) — discarding a roll also un-founds any den it created, so the world keeps nothing from dice you set aside. An **unnamed** side (a meeting you authored yourself, or one from an older save) offers the same roll and pick too — the panel names the table its hex calls for, and the first ⟳ rolls on it. Not every result is a monster: the same throw can turn up **civilized folk** — likewise named from the book’s table (bandits, pilgrims, herders, a merchant and his guards) — or a **terrain discovery** (a danger, a find, a wonder — their detail tables come with the encounter-content phase). Each meeting becomes an **Encounter** in the campaign — a real entity with two sides, a place, and a life of its own — created at the **rules-correct opening distance** for the terrain (a jungle meeting starts nearly on top of each other; on the steppe you see them coming from a quarter mile — RR pp.280–281, with visibility capped by light and how big each side looms). From there the engine walks the rulebook's pre-combat steps on demand: the **Surprise Matrix** (who knew, who sees — neither side aware means *no encounter at all*), **evasion** (terrain × party size, with the explorer/forlorn-hope/speed modifiers and the scatter-and-get-lost aftermath), the **reaction roll** (2d6 + the face's Charisma, with the natural-2/12 clamps), and **influencing reactions** — repeated parley attempts on the rulebook's time ladder (a round, ten minutes, an hour, a day, a week — the longer talks **cost the talker's day** on the Activities dashboard) with **bribes** (+1 to +3 by the target's pay, cheaper and safer with the Bribery proficiency, a step toward hostile when a bribe falls flat). Resolved encounters stay in the record — the world remembers who you met, where, and how it went. **Camps are finally dangerous:** a party resting in the wild now faces the rulebook's rest-and-stationary checks (every 12 hours day and night in unsettled wilds, nights in the outlands, rarer in settled lands — JJ p.41), surfaced through the Day Clock for the GM to resolve. **And so is hunting:** a day spent chasing game rolls the hex's encounter table once, as the rulebook prices it (RR p.278) — any meeting lands right on the Forage/Hunt result row with its own ⚔ Resolve button, and re-rolling an unlucky hunting throw never re-rolls the meeting. And you resolve every meeting in a **step-by-step encounter panel**, reached from one place: a new **World ▸ ⚔ Encounters** tab lists every encounter awaiting resolution (its badge counts them; the Day Clock tells you when committing a day has created one), with **past encounters kept in their own table beneath** — each with its outcome and the last attitude the other side showed — or click **⚔ Resolve** right where a meeting appears: a journey's day-log row, a search result, a lair's encounter list, or the hex card's **⚔ Encounters** card. In the panel you walk the rulebook's steps in order — **each phase appears only once the one before it is settled** (roll the distance and the awareness step appears; settle surprise and the evade-or-talk choice opens up, along with the ways to end the meeting — only the strike-it Dismiss is always at hand) — the opening distance; **who's aware of whom** (tick foreknowledge and line-of-sight per side and the panel names the outcome live — neither side aware means no encounter at all); the surprise dice — with a **Hidden** tick per side: tell the panel a side is lying hidden and the rulebook's consequences apply on their own (its opponents roll surprise at −2 and can't claim line of sight on it, so an ambush plays out by the book — RR pp.283–284; the Hiding skill throw itself stays the GM's call); **evasion** with the rulebook's modifiers laid out as switches (the explorer guide, the forlorn hope, flyers, the speed difference — and a surprised party can't run unless an unsurprised explorer leads it) — and **a successful escape plays out in full** (RR p.285): the fleeing party is displaced a rolled distance in a rolled direction and immediately makes its Navigation throw at −4, all arithmetic shown; fail it and the party is **lost — and knows it**: its journey **holds where it stands** (the travellers camp as a stationary group, still eating their rations, still facing the night's encounter checks) until it **searches for its last known landmark** — one button on the journey's panel, a search hour with its own encounter risk — or you re-draw its route; the **reaction roll** with your chosen spokesman's Charisma; and **parley** — each further attempt takes longer per the rulebook's ladder, and the panel shows what the next one will cost of the speaker's day before you commit to it, greying out when their day is full, with bribes priced automatically (cheaper and safer for a speaker with the Bribery proficiency). Every die in the panel shows its **full arithmetic** — the dice expression, each named modifier, the total against its target — and **every roll can be rerolled** while its step is still the live one (the opening distance until surprise is rolled, surprise until someone runs or reacts, a failed evasion until talk starts, the reaction until the first parley attempt, and always the latest parley attempt — a reroll keeps the same modifiers and never charges the speaker's day twice). End it as a parley, hand it to the table as combat, let it disperse, or strike it. Every parley also takes a **tone** — diplomatic, intimidating, or seductive (JJ pp.84–87) — and the panel lays out the rulebook's situational modifiers for the chosen tone as a checklist: what the campaign already knows it pre-ticks for you (the alignments, whose lair you stand in, who outnumbers whom — lair-mates counted, the monster's morale, your spokesman's proficiencies, and how you parted last time), the rest are your ticks and amounts, and every ticked line shows by name in the roll's arithmetic. Intimidation plays by its own rules: it can cow a creature (**intimidated** — slips away when it can, complies while it can't) or awe it (**overawed**) but never win a true friend, its gains are temporary, and the panel keeps the original roll on record for when the cowed creature's allies turn up (JJ p.86). A meeting can even **found a lair**: monsters met in the wild may linger and den right where you found them — their Lair % chance, doubled if an empty treasure-dungeon beckons; a second success settles them at full lair strength with a hoard, otherwise the band you met digs in with nothing yet (JJ p.69 + p.103) — but **never past the hex's maximum lairs** (JJ p.69: settled land supports only a third, half, or two-thirds of what the unsettled wild does — a full hex is simply too crowded, the offer says so, and clearing a den makes room) — under its own house rule, **Persistent wandering monsters** ("wandering-encounter survivors become placed entities on the world map"), on by default in the new **⚙ House Rules ▸ ⚔ Encounters** tab: the linger dice come from the rulebook's *domain*-encounter procedure, so applying them to wayside meetings is yours to switch off (off, the offer disappears) — the panel rolls it as a proposal you confirm — a roaming band from the Groups table settles **as itself**, the den adopting the band rather than rolling a fresh population — and the new lair appears under World ▸ 🐉 Monsters remembering the meeting that founded it; if the dice say they move on instead, the meeting closes as dispersed. The offer also stands **after your party evades**: the band you fled may still den behind you — the meeting stays evaded on the record, but a den founded there starts **unknown to your players** (they ran — searching the hex or tracking can find it later); one linger roll decides it, never re-rolled. And the offer explains itself when it can't fire — an unidentified "GM identifies" band reads *name the monster first*, a band already at its lair reads *they are already home*. **If your party has met these monsters before, the panel says so** — "These sides have met before — parleyed, last attitude indifferent" — drawn from the campaign's own encounter history, so a lair you've treated with remembers you (and one you've antagonised does too). Combat itself stays with the GM (record its result as an adventure outcome); the full creature-identity tables come with the combat phase.
- **Monsters can hunt down a party that fled them (optional).** A new house rule, **Monster pursuit** (on by default, in the new **⚙ House Rules ▸ ⚔ Encounters** tab — the rulebook leaves pursuit to the Judge, so the automation is yours to switch off): when your party evades a creature that can track (a wolf's nose, a hunter's craft), the encounter doesn't close — the panel asks whether it takes up the trail (the rulebook's Tracking throw, RR p.120 — easier against a numerous party), and that throw can be **rerolled** while the chase hasn't moved on: a lost trail re-found puts the pack on your heels (the evade un-resolves), a found trail re-lost ends it. A pursuer then follows at **half its expedition speed** through the Day Clock, rolling daily to hold the trail while the gap opens or closes with your party's actual movement — camp carelessly and it arrives; outpace it and the trail goes cold (rain, rivers, and *Passing Without Trace* are the GM's levers) — and if it catches up, a **fresh encounter springs at your party's hex**, with the chase on its record. The hunt's end isn't the band's: win or lose, a pack still standing **walks home to its den** — or wanders off as a migrant if it has none (see *Monsters now move in your world*). A hunting band is also **out in the world while it hunts**: a *different* party in the same hex whose wandering dice roll that monster meets *the* band — the panel says whose trail it's on (one click opens the chase) — and **scattering it ends the hunt** (your quarry's pursuit resolves behind it; parleying leaves the pack on the trail). A band mid-hunt won't den where you met it, and can't be tracked while the hunt runs — its motion belongs to the chase (the panel says so). Pursued characters show a 🐺 *stalked* warning on the Activities dashboard.
- **Your party can track anything it has met — and follow the trail until caught or lost.** Any band that **parted from a resolved meeting** — talked out, evaded, dispersed, or fought; with a den or without; monsters, migrating remnants, even civilized folk or a creature nobody has named — leaves a trail, and the **🐾 Track them** button on the meeting runs the rulebook's tracking rules in full (RR p.120). **Finding** the trail is the throw: Tracking 11+ after a turn's search (no retry for an hour on a miss), easier for a bigger band, with every printed modifier laid out in the dialog — soft or rocky ground (suggested from the hex, yours to override), the trail's age (−1 per 12 hours of good weather — the dialog shows how many days old it is), rain or snow since it was made, dim light, and extra Tracking ranks at +4 each. **Following needs no throw at all**: the trackers' journey is **steered after the quarry** — started on the spot, or your journey under way re-routed — at **half expedition speed** (the pace dropdown caps itself and says why), with no getting-lost navigation throws while the spoor leads, the destination updating day by day as the Day Clock walks both sides. The quarry **moves like a real thing in the world**: a band with a den heads home at full speed and goes to ground there; a den-less or migrating band **wanders** — half speed, drifting at random, never doubling straight back; folk make for the nearest settlement; a tracked roaming band from the Monsters tab moves on the map as the follow proceeds (and shows there as **tracked**, with whoever is on its trail). The trail **breaks only as the rulebook says**: when it enters water, or after an hour of rain or snow — tick *"rain/snow fell today"* on the meeting's 🐾 panel and the next day's step washes the trail out, forcing a fresh find at −4 per hour of rain (lost for good if it fails — searching can begin again from the meeting). **Catching up springs a real encounter at the quarry's hex** — and a band run to ground **at its den** is met at the lair, the whole den against you, with the arrival itself discovering the lair on your map; a band that settled in the very hex where you met it (or a fragment whose den is right there) is caught **at once**, no journey needed. Every meeting the follow springs remembers the one it came from. Give up any time from the meeting's 🐾 panel; trackers show a 🐾 *tracking* chip on the Activities dashboard.
- **Monsters now move in your world — migrating bands wander, and hunters walk home.** A monster band with no lair — a population that abandoned its den, the survivors of a broken hunt, any roaming remnant on the Monsters tab — now **wanders the map as the Day Clock advances**: half its expedition speed, drifting at random, **never doubling straight back** into the hex it just left (its Groups-table row says it's wandering; edit the band in the Inspector to park it). **A band that wanders into a domain is an incursion**: the day it crosses the border counts as that domain's daily encounter occurrence (recorded now, for the *Vagaries of Incursion* machinery that arrives with the mass-combat phase), and the band decides its disposition on the spot per the rulebook (JJ p.103) — **linger**, and it settles as a new lair right where it entered (unknown to your players, at full lair strength if the dice say so — the den adopting the band itself), or **migrate** and keep moving — and a hex already at its **maximum lairs** (JJ p.69) is simply too crowded: the band never lingers there, the entry still counts, and it moves on to another hex. And **a chase has an aftermath**: a pursuing pack that loses the trail — or catches its quarry and lives to part (scattered is another matter) — no longer vanishes; it **turns for home**, walking back to its den at full expedition speed without stopping for anything (a third party can still run into it on the way — and if a fresh hunt springs from that meeting, it chases, then resumes the walk home), dissolving back into the den's population on arrival; a hunter with no den to return to becomes a migrant and wanders. The Day Clock's review lists every band's movement day by day; it is all part of **Persistent wandering monsters** (⚙ House Rules ▸ ⚔ Encounters — switched off, the world stands still as before).
- **A miserable domain's bandits are now real bands in your world.** The rules have always said it (RR pp.350–351): a domain whose morale falls to **Turbulent (−2)** or worse is *plagued by bandits* — one able-bodied man per five families turns bandit at −2, one per two at −3, every able-bodied man in open rebellion at −4 — and the tool has long shown that count on the domain panel. Now they **take the field**: each monthly turn musters the domain's bandits as real bands placed on its own hexes (split across the domain so a travelling party meets a warband, not a statistic), visible as **🏴 rows on the Monsters tab's Groups table** and on the domain panel's bandit box (with a one-click jump to them). They are the domain's *own men*, and they behave like it: they **raid within their domain** as the Day Clock advances (roaming hex to hex but never leaving — they are not migrating monsters, and they never settle a lair: the encounter panel's den offer explains that they'll melt back to their fields instead), a wandering encounter in their hex **meets the band itself** (the panel names whose men they are), and meeting them runs the full encounter procedure — parley, evade, track them, or fight. And if the realm had trained, equipped **militia**, the rebels are drawn from them first (RR p.433): instead of peasant rabble you face your own heavily-armed, well-trained soldiers — those bands fight at their trained troop type's battle rating (far tougher than 1-HD bandits), the domain panel flags how many of the rebels are your militia in revolt, and only the surplus beyond the militia are rabble. The books balance the way the rules say: **bandits you kill are families you lose** (a band's recorded casualties settle at the monthly turn as population loss, RR p.351 — taking prisoners and restoring them to the land is yours to adjudicate), while **raising the domain's morale back to −1 disbands them with no loss at all** — the men simply return to their fields. And left unchecked they bite: bandits **count as an enemy army**, so each month the plague continues, the rulebook's occupation penalty builds on the domain's morale roll (nothing the first month, then −1 per month, cumulative — RR p.349), the growing drag shown among the roll's modifiers and on the bandit box. It's all under a new house rule, **Domain banditry takes the field** (⚙ House Rules ▸ ⚔ Encounters, on by default; off, the count stays a readout as before). And now you can **defeat the bandit army to heal the domain** (RR p.351): when your garrison — sallied out from the domain's Military tab, or set against the band in the battle engine — defeats a banditry band **in open battle, or drives a weaker band off the field without a fight**, the domain's current morale **rises by one** on the victory (the rulebook grants the heal for *successfully defeating* the bandits, and routing them off your land counts just as a pitched battle does — a bloodless rout costs no families, and the scattered men melt back to the population). The **prisoners you take in battle are your own men** — freed, they return to work (so only the bandits *killed* cost the domain families; you earn no ransom freeing your own peasants). If morale is still below −1 the freed men may turn bandit again next month — the rulebook's **Anárion** example reproduced exactly (a 1,600-family domain at −3 fields 800 bandits; defeat them, 400 slain and 400 freed, and it recovers to morale −2 and 1,200 families, then re-musters 240 the next month). The battle's **⚖ aftermath** shows the morale heal and the freed prisoners before you apply it. And when a domain festers, **an NPC bandit-leader rises from the ranks to challenge the ruler** (RR pp.350–351): each month at morale −2/−3/−4 there is a cumulative **1% / 5% / 10%** chance a leader emerges — a real NPC, his level set so his personal authority is exactly **+0** (RR p.350), who takes command of the bandit bands and offers battle at the first opportunity. **Meet him in battle and beat his army to break the challenge**; leave him unopposed and the next month he **loots and pillages the domain (−4 to its morale rolls)** until you do; or **raise the domain's morale above −2** and he loses his army and slips away. The domain's bandit box names him, with a link to his character sheet.
- **Hexes can now carry a climate, a terrain sub-type, and a biome — RAW's full terrain detail.** Beyond the ten base terrains, a hex now records an optional **terrain sub-type** (sandy vs rocky desert, deciduous vs taiga forest, forested/rocky/snowy/volcanic mountain, scrubby vs forested swamp, and so on), a **Köppen climate code** (the 30 codes from the *Climate by Terrain* table, JJ p.40 — the same value that governs weather, JJ p.41), and a **biome** derived from that climate. The hex editor gains a **Climate** picker that, once set, **suggests the matching terrain + sub-type** with one **✨ apply** click (the Köppen-led way to author a hex), a contextual **Sub-type** dropdown (shown only for terrains that have variants — a desert or mountain has them, open water doesn't), and a derived **Biome** readout (with an override for the rare case you want it to differ from the climate). The map's **Add/Edit hexes** brush carries the same **Sub-type** and **Climate** controls as new-hex defaults, so hexes you stamp on the map get them from creation (and editing a hex on the map loads them back) — and the **Sub-type** picker is offered there from the start (before you've chosen a terrain it lists the full set, narrowing to that terrain's variants once you pick one). A hex that has a sub-type is **named with it** throughout the tool — e.g. *Barrens (tundra) (0706)* — while a hex with no sub-type (the dropdown reads "unset") names as before. The sub-type makes wilderness detail RAW-exact: seeding a hex's lairs now rolls the **right density for that variant** (JJ p.69 — a forested mountain teems where a rocky one is sparse; the Lair Wizard's seed preview reflects it), and the same sub-type will drive line-of-sight (RR p.275) and the monster-encounter tables (JJ pp.45–67) as those land. Everything is additive — existing campaigns and the bundled templates load unchanged, and a hex with just a base terrain behaves exactly as before. *(Climate-correct weather and the encounter tables that read these come next.)*

### Changed
- **A journey's encounter check is now the rulebook's, hex by hex.** Travel used to make one flat one-in-six check per day, and roads suppressed encounters entirely. Now every hex the party **enters** gets its own throw on the real territory table (JJ p.41: one throw per 6-mile hex) — a day that crosses four hexes risks four meetings, each judged by *that* hex's territory — and a road makes the day **safer, not silent** (the +Road column trades monsters for civilized traffic). Hexes you've already traversed don't re-offer terrain discoveries (JJ p.42), and the day's meetings land in the day log and pause the Day Clock for the GM exactly as before.

### Fixed
- **The Inspector's ✨ Create button now opens the right creation form.** Picking a kind and clicking ✨ Create always showed the "no dedicated wizard yet" placeholder, even for kinds that *have* a dedicated creator — so Create → Character / Domain / Journey / Lair never launched their wizard. The button now dispatches to the kind's creator (e.g. the Lair Wizard); kinds without one still show the placeholder as before.

## [0.23.0] - 2026-06-08

### Added
- **Vassals now answer to their lords month to month — Favors & Duties.** Each month, every vassal domain receives one edict from its liege, rolled on the Favor/Duty table (RR pp.345–348): a **duty** demanded (build forts, pay scutage, loan gold, send troops, attend council, muster an army) or a **favor** granted (a gift, a charter of monopoly, a ceremonial office, stationed troops, a grant of land) — or a previous edict revoked. The lord may **safely demand one duty a month, plus one more for each favor he's granting**; pile on more duties than that and the vassal's loyalty is tested on the Hireling Loyalty table, at a growing penalty for each extra duty (RR p.347) — the mechanical heart of holding a realm together. The gold moves where the rules say it does: **scutage** is a monthly tax the vassal sets to **pay automatically** with a **💰 Pay Scutage** button on their own card (and a **🛑 Stop Paying** button to halt it) — once turned on it settles every month on its own until stopped, and the lord's card shows at a glance whether it's **being paid**. The amount is **1 gp per family in the vassal's realm** (a per-family rate you can lower to demand less), **recomputed as the realm grows or shrinks** rather than frozen at a fixed sum; what's paid counts as the vassal's own garrison expense (RR p.347), and a lord who collects scutage but doesn't out-spend it on troops provokes his scutage-paying vassals' loyalty rolls (RR p.348). A **loan** is *demanded* by the lord but **given** by the vassal — no gold moves until the vassal hands it over with a **💰 Give loan** button on their own card (the lord's card shows a "⚠ not yet given" notice meanwhile), and once given it is repaid to the vassal when the lord revokes the loan, or by the lord's Charisma rolled each month (RR p.348 — repayment lapses the duty, no interest); a **gift** flows from lord to vassal (a one-time favor — it lapses on the next monthly turn, so it never lingers past its month); a **construction** duty lets the lord **order structures** — either **generic** ("build anything, anywhere in the realm") or **specific**: pick a hex in the vassal's realm and a type (bridge, road, fort, tower, other structure, or — on a coastal realm — a vessel), add as many as you like, and add more later while the duty runs; each specific hex raises the build target by 15,000 gp while a generic order targets the whole realm (RR p.348), and the duty's card shows how much has been built and how much of the target remains. An **office** favor grants the vassal a ceremonial title you write yourself (e.g. "Knight Marshal") and, per RAW, gives the holder's own vassals a +1 to their loyalty rolls (RR p.348); if your realm is a senate, the owed senate seat is noted for when senates arrive. (It spends the vassal's coffers each month toward that target for now; detecting the vassal's actual building comes with the full construction system.) A **Call to Council** duty names a place in the lord's domain (defaulting to where the lord is now) the vassal must attend: his card gains a **🧭 Go to Council** button that sends him — or his whole party — there as a journey, plotting a new one or re-routing one already under way, and the lord's card shows at a glance whether the vassal is **en route** or has **arrived**. Duties whose effects belong to systems still to come (mustering armies, senate seats, monopolies, new domains) are recorded with a note for you to resolve in play. It's all driven by the monthly **Advance Month** — on by default, RAW as written; turn off **"Favors & Duties — auto-roll the monthly edict"** in ⚙ House Rules to hand-author edicts instead. A new **Favors & Duties** panel on each domain's **Vassalage** tab shows, for every vassal, how many duties are in force and how many the lord may still safely demand (with a warning the moment he over-demands), the favors and duties currently in force (each muster with its schedule), and one-click **Grant favor**, **Demand duty**, and **revoke** controls — so you can raise or lift an edict by hand at any time, demanding too much rolling the vassal's loyalty just as the monthly edict would (a vassal's own tab shows what its liege has imposed). When you demand a gold-bearing duty you can **dial the amount down from the RAW maximum** — a lord may always demand less, a half-sized loan or a partial scutage (RR p.345) — and a **✏️ Custom…** option records a **freeform favor or duty of your own devising**, with an optional one-time or every-month gold flow, for edicts the table doesn't cover. The 🔍 Inspector can also create and browse them. Every edict is logged to the Event Log.

### Fixed
- **Confirmation toasts now actually appear.** Several pop-up confirmations — adding/splitting/merging a garrison unit, emitting an event, and adding or editing a Chronicle entry — were calling the wrong internal function and silently did nothing (and could throw if another toast was already on screen). They now show as intended.

## [0.22.0] - 2026-06-08

### Added
- **Characters can forage and hunt to live off the land.** A new **🌿 Forage / 🏹 Hunt** action — on a character's new **Survival** tab, on any hex card, and as a quick-action on the Activities dashboard (beside Market / Recruit / Travel) — lets a character gather firewood, water, or food, or hunt game — the camp-and-trail counterpart to shopping at a market. Foraging takes one short part of the day; hunting takes the whole day. The **Survival** proficiency adds +4, and terrain and territory shift the odds for food and hunting (game is scarce near towns, plentiful in the deep wild); water is free and automatic wherever there's a river, lake, or settlement. A success tops up your water, adds days of food to the pack, or yields firewood — and the effort shows on the Activities dashboard and counts against the character's activity budget (RR p.278). Attempts **stack** in the action (forage several things in a row), and **every throw has a one-click re-roll** that swaps the result and its yield. The action **respects the character's day-activity budget**: an option greys out — with the reason — when there's no time left for it (foraging is a short task, hunting takes the whole day; water at a river or settlement is free).
- **Food and water are now tracked for any character out in the wild — not only travelling parties.** When you advance the Day Clock, a character standing in the wilderness eats from their pack and drinks their water (and goes hungry or thirsty if they've run out), exactly as a journeying party does. A character who is in a settlement, at their stronghold, or **in a realm they rule — or one of their vassals' realms — is fed and watered automatically** (food and lodging there is part of their cost of living) and leaves with full waterskins; **and that lifestyle covers the companions travelling with them** — a lord's henchmen and hirelings don't touch their own rations while marching through his march, and a journey that crosses your own domain spends nothing. Out in country no one holds, the party lives off what it carries (always — a party that finishes a journey and camps in the wild keeps needing food and water). Each character sheet gains a **Survival** tab — food and water stocks, current hunger/thirst with Constitution lost, whether they're living off the land or off their lifestyle (and why), and the Forage / Hunt action — and the Journey and Parties panels show a **🏠 On lifestyle — no food or water consumed** banner whenever the party is somewhere that provisions it. When the Day Clock surfaces a hungry or thirsty character, the day-tick review now **names them and shows the detail** — who, where, their hunger/thirst stage and the Constitution lost, whether they ate and drank, and the food and water they have left — with a **Survival →** link to step straight to that character and forage or move (field survival always applies, so there's no "reject" — you act on it instead). A party can be set to **share provisions** so its members pool food and water whether they're travelling or not. A day on which a field character goes hungry or thirsty — or recovers — is now **written into that character's history and the hex's history**, so you can look back at what befell someone in the wild (routine fed days aren't logged, and a recovery stays out of the narrative Campaign Log). When the Day Clock surfaces a dehydrated character, the Constitution loss shown in the day-tick review is **stable** — re-opening the review, or cancelling and ticking again, shows the same outcome instead of re-rolling a fresh one each time (it changes only when you advance the clock or get the character water). (The settled side of this — the monthly cost-of-living rule — is below.)
- **Characters now pay monthly living expenses, and what they spend sets how important they look.** A new **Living expenses** house rule (on by default — turn it off in ⚙ House Rules) charges each self-supporting character a monthly cost of living equal to the wages of their level (RR p.173: ~25 gp at level 1, 800 at level 6, 7,250 at level 9), paid at month end from their coin purse — or, **for a domain ruler, from the domain treasury by default** (a ruler funds his own lifestyle and his henchmen's wages from the realm's coffers; untick the option on his Expenses tab to pay from his personal purse instead — a character who rules no domain always pays from the purse). You can dial a character's **lifestyle target** up or down on their new **Expenses** tab; they pay the lesser of that target and the coin they actually have, so being short of funds simply means spending less — never going into debt. **Spending too little makes a character look less important to NPCs** (RR p.173): underspend and you're taken for a lower-level figure — which lets you retain fewer henchmen and makes the ones you have more likely to question your authority. (Spending *above* your level doesn't make you look grander — RR p.173 only adjusts apparent standing downward; a lavish bluff to impress a henchman is left to the GM.) The character sheet gains an **Expenses** tab that gathers all of this in one place: the apparent level beside the true one (and why), the lifestyle-target lever, **a table of the henchmen and specialists the character pays wages to** — each with their level and monthly wage, with a **Total henchmen wages** line (a vassal-ruling henchman whose land covers his own wage is shown as *waived*) — and a **Total monthly expenses** line (lifestyle keep + wages); the **Advance Month** review previews every character's keep before you commit. **Lieges also pay their henchmen's and specialists' monthly wages** in the same month-end pass (a henchman given a vassal domain that earns at least his wage is paid by the land instead). The Loyalty-roll dialog now pre-fills the apparent-level penalty when a henchman outranks his employer's apparent standing.

### Changed
- **Recruiting hirelings is now a day-by-day search that takes the patron's time.** Soliciting for henchmen, specialists, or mercenaries is no longer an instant "spend a week" click — it's an ongoing **search** that runs while time passes. You **Start soliciting** in a market, and as you advance the **Day Clock** candidates trickle in over the following weeks — about half **after** the first week, a quarter after the second, the rest after the third (RR p.164) — and each solicited week charges its fee from the patron's purse. The search panel shows how many days into the current week you are (e.g. *day 2/7*) so you can see the next batch coming. While a search is running it occupies one of the patron's short daily tasks, so it shows on the **Activities dashboard** and the character's sheet — every day it runs, **including the final day** of the three weeks, when the last candidates arrive (that day is still soliciting, not a separate hiring day) — and you **can't start one when the patron's day is already full** — an administering ruler, for instance, has no time to canvass the taverns (the same activity budget that gates foraging and shopping). Engaging, negotiating, and hiring candidates work exactly as before, now drawn from whoever the search has turned up so far; submitting your hires concludes the search, or **Stop soliciting** ends it early. ### Changed
- **Items in stashes and carry inventory now use one composable shape.** Instead of a fixed coin / bulk / item split, an item line carries a set of toggleable *facets* — coin, valuable, gear, bulk, magical, readable, container — so a single line can be several things at once (a jewelled, enchanted blade is gear + valuable + magical). Coins can be any denomination (copper, silver, electrum, gold, platinum) and a line's weight and gold-piece value are computed automatically — coins weigh 1 stone per 1,000 (RR p.83), and gems and jewelry carry a per-piece value. A mundane line can be **promoted** into a tracked, named item with its own history once it starts to matter (the same "the world remembers" pattern as recurring monsters becoming lairs). Existing campaign files upgrade automatically on load. (Groundwork for the Stashes view, the character Vault card, and treasure — landing next.)
- **Encumbrance is now modelled by the book, by default.** The optional encumbrance house rule used to be a default-off *hard block*; it is now **Ignore carry encumbrance** (still default off). Left off — the default — a character's carry weight and load band are computed per ACKS (coins weigh 1 stone per 1,000; unencumbered through overloaded, RR pp.83–84) and you're never stopped from picking things up: being overloaded just means you can't move. Turn it on to tell the tool to ignore weight entirely. (The on-sheet readout arrives with the Stashes UI.)

### Fixed
- **Editing a settlement's Families or Investment on an older campaign file no longer silently reverts.** Settlements saved before they were given stable internal IDs couldn't be edited — the value would flash and snap back (with a "no entity to save against" warning in the console). Loading a campaign now assigns any such settlement an ID, so its Families, Investment, and other fields save normally. Existing saves are repaired automatically on load — just reload the file.

## [0.15.0] - 2026-06-03

### Added
- **Journeys now travel hex by hex, following the roads and rivers you've drawn on the map.** Until now a journey moved by straight-line distance and read only a single coarse "this hex has a road" flag on its *starting* hex. It now walks the actual line of hexes between start and destination and resolves each one against the per-side **roads**, **rivers**, and **ford/bridge** marks from the map's Add/Edit-hexes editor:
  - **Roads speed you up and keep you from getting lost** (RR p.272 / p.275). A hex you pass *through* earns the road bonus (×3/2 speed, no getting-lost throw, the safe wilderness-encounter column) when a road connects the side you enter by to the side you leave by; a hex you *end* in counts when a road touches the side you entered. A day spent entirely on roads rolls no navigation throw and draws no wandering encounter.
  - **Rivers are a barrier you have to cross** (RR p.271). Reaching a river edge with no ford or bridge forces a Swimming throw: make it and the party fords across — but the crossing takes the rest of the day (you're swimming, not marching); miss it and they're held at the near bank and the day pauses for you to resolve the crossing (the drowning risk RAW spells out). A **ford/bridge** mark — or a **road that crosses the river** (an automatic bridge) — lets the party cross for free.
  - **Terrain now varies along the route.** Each hex's own terrain (and any mud or snow underfoot) sets the pace for that leg, so a trek that starts in grassland and crosses a belt of forest or jungle slows down exactly where the hard ground is, instead of treating the whole trip as the starting hex's terrain.
  - **Watch the party move across the map.** The Journey Detail day log now lists the hexes crossed each day — named where you've drawn them, shown by hex number where you haven't — and marks where the party is **right now** (📍). And the travellers themselves (and their party) are **placed at the hex they've reached** as the journey advances, not moved only on arrival — so the roster and map show them progressing along the route. (They settle at the nearest hex you've actually placed; across un-drawn stretches they hold at the last one until they reach the next.)

  Where you haven't drawn a particular hex along the way, the journey falls back to the starting hex's terrain and road state — so **existing campaigns travel exactly as before**, and the new detail rewards filling in the map. (The per-side road/river/ford cartography shipped in 0.13.0–0.14.0; this wires it into travel.)
- **Pick a journey's destination — and its waypoints — straight off the map.** The Start-a-Journey planner has **🗺 choose from map** links for both the Destination and the Waypoints: jump to the map, click the hex(es) you want, confirm, and land back in the planner with everything filled in (and everything else you'd entered left intact). The **destination** is a single click (re-click to change your mind). **Waypoints** are multi-click — click hexes in the order you want to visit them, click one again to drop it — and they show in the planner as ordered, removable chips instead of the old checkbox list. As you pick, the **map draws the route live**: start in green, destination in red, numbered waypoints in between, joined by a dashed line — so you watch the path take shape, and the preview shows whether or not the Journeys map layer is switched on. Handy when you know *where* places sit on the map but not their hex numbers.
- **Change a journey's destination or waypoints mid-trip — on the map.** A journey already under way can be re-routed without starting over. Open it and click **🗺 View on map** to see its route drawn and the view centred on it (the journey you're viewing stays visible even if you've switched the Journeys map layer off), then use **✎ Change destination** or **✎ Change waypoints** right there — click the new hex(es), watch the route redraw, and confirm. The party keeps going **from exactly where it is** (it doesn't jump back to the start), the journey still remembers where it originally set out from, and re-routing also gets a lost party back on a heading. (Side effect of the same work: a journey with waypoints now travels its *whole* route through each one, instead of cutting the corner once it had covered the straight-line distance.)

### Changed
- **Activities now opens on Journeys.** The Activities tabs lead with **Journeys** (then Ventures, Hijinks, Spell Research) instead of Ventures, and opening Activities lands on the Journeys panel — the active day-aware subsystem — rather than the Ventures list.

## [0.14.0] - 2026-06-03

### Added
- **Create Map — lay out a whole grid of blank hexes at once** (a ➕ Create Map button on the 🗺 Map tab, and the first thing you see on an empty map). Pick a size — 10×10, 25×25, 50×50, 100×100, or any custom width × height — and the tool fills in that many blank hexes (no terrain, unexplored, unclaimed), numbered in the published column·row convention from `0101`, ready to paint a world onto. Hexes you already have are **kept, never overwritten** — the grid fills in around them — so you can also use it to extend an existing map outward. An optional starting column/row places the block wherever you want, and a live preview shows how many hexes it will add (and how many already exist in range).
- **Reassign a hex to a different domain from the hex detail panel.** The hex popup now has a **Domain** dropdown — every domain, plus *Unclaimed wilderness* — so you can move a hex between realms (or release it to the wild) without leaving the panel; it re-homes the hex, re-anchors the open panel to it, and is recorded in the event log like every other edit. Previously this was only possible from the map's Add/Edit-hexes mode.
- **Delete a hex from the World ▸ Hexes list.** Each row's Actions column now has a **delete** next to **edit**. It removes the hex entirely — from the map and from its domain — and takes its settlement with it; if the hex holds a settlement, lairs, dungeons, or points of interest, the confirmation names what will be lost first. (To merely take a hex *out* of a domain without deleting it, set its **Domain** to *Unclaimed wilderness* instead.)
- **Create and see unclaimed (wilderness) hexes in World ▸ Hexes.** The “+ add hex” control now offers *Unclaimed wilderness* alongside the domains, and the Hexes list shows domainless hexes too — it aggregates the whole campaign, not just owned hexes — so you can build the map out between and beyond your domains and claim a hex later via its Domain field. Unclaimed hexes start **unexplored** (uncharted until you survey them).

### Changed
- **The map's Add/Edit hexes panel now opens the moment you turn the mode on** — you no longer have to click a cell first to see it. With no hex selected it sets the **defaults for the next hex** (domain + terrain): choose them, then click an empty cell to place a hex with those settings (they carry over instead of resetting), or click an existing hex to edit it. Rivers/roads and Save appear once a cell is selected. "Done" leaves the mode.
- **"+ add hex" now places the hex on the map.** Both the per-domain Hexes table (Domain ▸ Demographics) and **World ▸ Hexes** "+ add hex" now open the Map in Add/Edit mode with your chosen target — a domain, or *Unclaimed wilderness* — pre-selected in the usual hex picker; click an empty cell to place the hex and you're returned to where you started (the domain, or the Hexes list), with cancelling bringing you straight back too. (On a still-empty map, where there's nothing to click yet, it falls back to creating the hex inline.)
- **The hex panel's Terrain field is now a dropdown** (the same terrain catalog the map's Add/Edit-hexes editor uses) instead of a free-text box, so terrains stay consistent with the map's colouring and travel rules. A hex whose terrain isn't in the catalog (e.g. an older "plains"/"coast") keeps its value — it's offered as a "(current)" choice and stays selected — and "— none —" clears it. The change is logged like any other hex edit.
- The per-domain **Hexes** table (Domain ▸ Demographics) is tidier: the row-level **×** delete column and the trailing **World / edit ↗** column are both gone. The hex's coordinate is itself the link that opens the hex detail panel, so the separate edit link was redundant; and to take a hex out of a domain you open it and set its **Domain** to *Unclaimed wilderness* (which keeps the data consistent, unlike the old delete button).

### Fixed
- **A hex that belongs to a domain is no longer treated as unclaimed after loading.** Hexes store their domain in two places — a canonical `domainId` and each domain's hex list — and on load the tool backfills the first from the second. That backfill could miss the surviving copy of a hex when the campaign file (or the in-browser session cache) already listed it at the top level without a `domainId`, so the hex would show as unclaimed grey on the map's domain colouring and drop out of domain hex filters even though it sat squarely inside a domain. Loading a campaign now backfills the domain association onto the copy that's actually kept, and opening a hex whose association is still missing repairs it on the spot from its domain's hex list.
- **A hex's coordinate now reads the same in the editor as on the map.** The hex editor's coordinate field showed the internal axial `(q, r)` while the map and the hex name showed the GM-familiar column·row hex number — so the same hex appeared to carry two different numbers (a hex at column 151, row 099 displayed as "151174"), and a column·row number typed straight into the editor landed in the wrong place. The editor (and the Inspector's coord field) now work in **column·row** — the convention published Auran/JG maps use, and the one the map labels already showed — with the internal axial value shown on hover. Type a column·row hex number — leading zeros and all (e.g. "099") — and it reads back the same and sits where it should. Hex numbers now read as a zero-padded, run-together column·row — `151099` (column 151, row 099), each part padded to a uniform width so the number splits cleanly into column and row — rather than the ambiguous `15199`. The remaining places that still showed a bare `(q, r)` — settlement and improvement tables, journey destinations, character-location pickers, hex search — now read column·row too. Axial stays the internal store, so existing campaign files load unchanged.

## [0.13.0] - 2026-06-02

### Added
- **Campaign map — a clickable, zoomable, layered hex map of the world** (a new 🗺 Map view under the World tab). Every hex renders as a flat-top hexagon in its true position, labelled with a RAW-style column-row coordinate (RR p.273); **scroll to zoom, drag to pan**, and **click a hex to open its existing card**. Three kinds of layer build the map you expect:
  - **Color by** (one at a time, with a legend) — **terrain** (ten base types — nine land plus **open water** for oceans, seas, and big lakes), **domain** (a stable colour per realm; unclaimed grey), **land value** (3–9 gp/family heatmap, RR p.341), **classification**, **population** (vs the RR p.340 ceiling), **domain morale**, **secured** (stronghold adequacy, RR p.338), **economy**, and **exploration**.
  - **Symbols** (toggleable) — settlements as circles sized by market class (RR p.351), strongholds, and lair / dungeon / point-of-interest markers.
  - **Edges** (toggleable) — computed **domain borders**; **rivers** drawn along hex edges; **roads** running from a hex's centre out to its sides (bending through the centre with a faintly circular curve); **trails**; and **ford/bridge** marks where a river is crossed. Active **journeys** show their route as a highlighted path with a marker at the party's current hex.
  - A one-click **Standard view** sets the familiar composite, and an **Add/Edit hexes** mode lets you click an empty cell to **create** a hex or click an existing hex to quickly **edit** it — setting its domain (including assigning an unclaimed wilderness hex to a domain), its terrain (now including open water), and its **rivers, roads, and crossings**: on a small hex diagram, click a side to run a **road** out from the centre, or cycle a side through **river → river + ford/bridge → none**; a road that crosses a river is automatically bridged. The first time the map reaches *and* shapes the lands between domains.

  It is a pure view of the hexes you already have — nothing new is written to your campaign file — and the first surface that makes the whole world legible at a glance.

### Changed
- **The World tab is now a hub for Map, Domains, and Hexes.** Map, Domains, and Hexes are sub-views under World (in that order) rather than separate top-level tabs — Domains, previously its own top-level tab, now lives under World ▸ Domains. The tool still opens on the domain manager (now World ▸ Domains), and which World sub-view you were last on is remembered across reloads.
- **Hexes now have a consistent name everywhere they're referred to** — `<Settlement> (coords)` if the hex has a settlement, otherwise `<Terrain> (coords)` (e.g. "Saltspur (0000)", "Forest (0100)"). It shows in the hex card, World › Hexes, the Inspector, journeys, and character locations — replacing the bare `(q, r)` coordinates that appeared before. (The terse number at the top of each hex on the map is unchanged.)

### Removed
- The unused `hex.riverCount` field. It was never wired to anything and cited a movement-cost rule that doesn't exist in RAW; rivers are now modelled as the edges they run along, with ford/bridge crossing marks. (Dropped automatically when a campaign loads — no action needed.)

## [0.12.0] - 2026-06-02

### Added
- **Journeys panel — plan and run overland travel from the interface** (Activities › Journeys). A guided **Start a journey** action picks the travellers (individual characters or a whole party, with the mercenaries that travel with them, RR p.166), a start and destination hex (plus optional waypoints), and starting rations — with a distance-and-days forecast before you commit. A **Journey Detail** panel shows the route, a day-by-day log (each day's navigation throw in full, with a GM **reroll** on the current day), a supplies tracker with low-stock warnings, a **Members** table (each traveller, their survival conditions, and their mercenaries), and **Tick day** / **Abort** controls; **pace and travel mode** are set on the journey itself (and can be changed mid-journey). **Send on journey** shortcuts sit on the character sheet, the party panel, and the hex card (which also lists journeys passing through it). Advancing a journey runs through the shared **Day Clock** — the same propose → review → commit as construction — so travel stays in lockstep with the calendar, and it is RAW-accurate: pace is **Normal / Half speed (×½) / Forced march (×1½, fatigued at once)** (RR pp.272, 279) and survival reads **Hungry → Underfed → Starving** with a single **Dehydrated** stage (RR p.276). (The travel engine shipped in 0.11.0; this release makes it hands-on.)
- **Parties panel — founder-based parties you can send travelling together** (Characters › Parties). **+ New party** picks a founding character; the party is named after them, placed at their hex, with the founder as the first member and leader. Members are added from the **same hex** (so a party gathered in one place can set out together); the leader is marked ★ and reassigns automatically, or by **make leader**; and the party's name, notes, and current hex are **lock-edit** fields logged on save. Member names open the character sheet, and each member's **Mercenary Company** (RR p.166) rolls up into a party headcount total.

### Changed
- **Ventures: eligibility, a clearer ETA, and a labelled GM override.** A character already away on a venture (or a journey) is now **hidden from the New-venture and Start-a-journey pickers** until they return (RR pp.370–380 — one active venture per character); their sheet shows a **🛒 On venture →** link instead. The **Turns** column now reads the **time until arrival** ("in 2 turns" / "ARRIVED") rather than an absolute turn number, and while a venture is still in transit its arrive action reads **GM-arrive** (signalling a forced early arrival), flipping to **→ arrive** once it's due.
- **The day-tick review reads more clearly.** The "Paused for review" note now appears **only when a multi-day advance (a week / month) actually stopped early** — a single **+1 day** no longer shows it (the review always opens for one day) — and when it does, it says how far it got. With several day-aware activities in flight, each one's warnings (lost / hungry / dehydrated / encounter) are listed beneath **its own** record rather than in one shared pile.

### Fixed
- **Overland travel speeds and getting-lost throws now match the rules as written** (RR pp.272–278 / JJ p.38). **Fog and snow halve** travel speed, and **frigid or sweltering** temperature halves it; **mud or snow** underfoot halves it; **rain and storms do not reduce base travel speed** (a storm is an activity-penalty condition, not a speed reducer — RAW has no ×¼ weather); and the worst terrain is dangerous to navigate again — **dense scrubland** (8+) and **forested swamp** (14+) carry their tougher getting-lost throws instead of their easier cousins'.
- **Editing a per-hex family count (or the domain-level Peasant families field) no longer throws an error and reverts the edit.** The internal sync that keeps the domain total and per-hex counts in agreement was calling engine helpers that weren't reachable from the event module; it now routes through the exported setters.

## [0.11.0] - 2026-06-01

### Added
- **Groundwork for overland travel** (engine + data layer — there's no Journeys UI yet). Under the hood, the engine can now model a journey day by day — movement, navigation, rations, and fatigue (JJ p.84), advanced by the Day Clock. You can't plan or run one from the interface yet; a dedicated Journeys panel comes in a later release. This release just lays the foundation it will build on.

### Changed
- Internal travel logistics default to RAW (rations + the six-day fatigue cycle, JJ p.84); the `simplified-fatigue` and `ignore-rations` opt-out house rules are reserved for when the travel UI ships.

### Removed
- Two unused placeholder house rules (`realistic-fatigue`, `mandatory-rations`), folded into the RAW-default approach above.

### Fixed
- **House-rule toggles now take effect for rules stored as a bare boolean.** Some rules (e.g. families-per-hex tracking) ship in templates as `true` rather than `{enabled: true}`; the House Rules UI only understood the object form, so those rules displayed as off and their toggle wouldn't flip. The UI now uses the engine's canonical accessor (which accepts both shapes) and normalizes the value when toggling. This is what made families-per-hex tracking look broken — the per-hex columns never appeared even with the rule "on".
- **Families-per-hex tracking** no longer loses per-hex edits. With the house rule on, the hexes are the source of truth: the domain's family total is now derived from the sum of its hexes, instead of per-hex family counts being silently rescaled back to the old domain total on reload (which also threw off the next monthly population-growth roll). Editing a hex's families now updates the domain total immediately.
- A disabled (`{enabled:false}`) **Markets Transaction Threshold** house rule no longer fires its notable-transaction rumor when a treasury grant lands — the gate now honours the rule's enabled flag rather than the mere presence of the setting.
- Event application is now **transactional**: an event whose handler errors part-way through (e.g. a complex adventure result granting treasure, XP, and casualties) rolls back cleanly instead of leaving partial changes behind, and is still logged as rejected — with the campaign left untouched.
